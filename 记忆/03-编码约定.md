# 03 · 编码约定

> **摘要**：项目内代码风格、命名、注释、错误处理的统一约定。**写代码前必读**，避免"上一个 Agent 用 snake_case，下一个用 camelCase"。
> **更新频率**：季度级（约定一旦定下尽量不改）
> **最后更新**：2026-05-01（追加 A2 包管理 / A3 Symbol 命名 / B9 覆盖率门槛）

---

## 1. 语言与版本

- **Python 3.11+**（用 type hints、`match`、`X | Y` 联合类型语法）
- 不写 Python 2 兼容代码
- **包管理 / 虚拟环境**：`uv`（决策 A2，2026-05-01）
  - `pyproject.toml` 描述依赖（不用 `requirements.txt`）
  - `uv lock` 生成 `uv.lock` 入 git
  - `uv sync` 安装；`uv run pytest` 跑测试
  - CI 也用 `uv sync --frozen`
- 备选：`pdm`（与 uv 兼容的 PEP 621）；不用 poetry / pipenv

## 2. 命名

### 2.1 标识符

| 类型 | 风格 | 例子 |
|---|---|---|
| 模块 / 包 | 全小写 + 下划线 | `data_feed.py` |
| 类 | PascalCase | `StrategyManager` |
| 函数 / 方法 | snake_case | `compute_atr()` |
| 变量 | snake_case | `current_price` |
| 常量 | UPPER_SNAKE_CASE | `MAX_LEVERAGE = 3` |
| 私有 | 单下划线前缀 | `_internal_state` |
| 类型变量 | T 大写开头 | `T = TypeVar('T')` |

### 2.2 文件名

- Python 源码：英文 snake_case（避免中文路径在 import 出问题）
- 文档：中文（如本项目的 docs/、记忆/、skills/）
- 配置：英文小写连字符（`s1-btc-trend.yml`）
- 数据文件：英文（parquet/sqlite）

### 2.3 业务术语

| 概念 | 代码用词 |
|---|---|
| 标的 | `symbol`（不是 `coin` / `pair` / `market`） |
| 信号 | `signal` |
| 订单 | `order` |
| 成交 | `fill` |
| 持仓 | `position` |
| 时间戳 | `ts`（毫秒 UTC int） |
| 周期 | `timeframe`（缩写 `tf`） |

→ 这些词在代码、DB 字段、API 里**全部用同一个词**，避免混淆。

### 2.4 Symbol 命名规范（决策 A3，2026-05-01）

| 用途 | 形式 | 例子 |
|---|---|---|
| 对内统一（代码 / DB / parquet 路径 / 日志） | **无分隔符大写** | `BTCUSDT` |
| 对外展示（Dashboard / Telegram） | 带斜杠 | `BTC/USDT` |
| 跨交易所 / 跨产品消歧 | `(exchange, symbol, type)` 三元组 | `(binance, BTCUSDT, perp)` |

`core/data/symbol.py` 的 `Symbol` 工具函数负责所有规范化：
- `Symbol.normalize('BTC/USDT')` → `'BTCUSDT'`
- `Symbol.normalize('btc-usdt-swap')` → `('BTCUSDT', 'perp')`
- `Symbol.display('BTCUSDT')` → `'BTC/USDT'`（仅 UI 用）

不同 type（spot vs perp）同名时**必须** 带 type 字段消歧；DB `symbols` 表用 `(exchange, symbol, type)` 复合唯一约束。

## 3. 注释与文档字符串

### 3.1 函数 docstring（必写）

```
def compute_atr(prices: pd.DataFrame, window: int = 14) -> pd.Series:
    """计算平均真实波幅 (ATR)。

    Args:
        prices: 含 ['high', 'low', 'close'] 列的 DataFrame
        window: 平滑窗口期，默认 14

    Returns:
        每根 K 线的 ATR 值 Series

    Raises:
        ValueError: 当输入 DataFrame 行数 < window 时

    备注：
        - 用 Wilder 平滑（不是 SMA），与 TradingView 一致
        - 窗口不足时前 N 行返回 NaN
    """
```

### 3.2 行内注释

只解释**为什么**，不解释**做什么**：

```python
# ❌ 不要这样
i += 1   # i 加 1

# ✅ 这样
i += 1   # 跳过表头
```

### 3.3 中文注释

- 业务逻辑、决策依据 → **中文**（更准确）
- 函数 docstring 第一行可中可英，但**统一**：本项目用中文
- 变量/函数名 → **英文**

## 4. 错误处理

### 4.1 异常分类

| 类型 | 处理 |
|---|---|
| 业务错误（如信号被风控拒绝） | 返回带原因的对象，不抛异常 |
| 可恢复错误（如 API 限流） | 重试 N 次后再抛 |
| 不可恢复错误（如 DB 连接失败） | 立即抛，由 engine 层处理 |
| 编程错误（如类型错误、断言失败） | 抛 `AssertionError` / `TypeError` |

### 4.2 自定义异常

```python
class CryptoQuantError(Exception): ...
class DataFeedError(CryptoQuantError): ...
class OrderRejectedError(CryptoQuantError): ...
class RiskBlockedError(CryptoQuantError): ...
class ExchangeAPIError(CryptoQuantError): ...
```

→ 顶层捕获 `CryptoQuantError` 即可处理所有项目自定义异常。

### 4.3 日志而非吞异常

```python
# ❌ 严禁
try:
    do_something()
except Exception:
    pass

# ✅ 至少要 log
try:
    do_something()
except Exception as e:
    log.exception("do_something failed: %s", e)
    # 然后视情况：raise / return None / fallback
```

## 5. 日志

### 5.1 日志级别

| 级别 | 用途 |
|---|---|
| `DEBUG` | 详细执行流程（默认不输出） |
| `INFO` | 正常业务事件（信号/订单/成交） |
| `WARNING` | 异常但可恢复（API 重试、轻微滑点） |
| `ERROR` | 业务失败（订单拒绝、风控触发） |
| `CRITICAL` | 系统级问题（DB 挂了、熔断触发） |

### 5.2 结构化日志

用 JSON 格式，方便后续接 Loki/Grafana：

```python
log.info("signal_placed", extra={
    "strategy": "s1",
    "symbol": "BTCUSDT",
    "side": "long",
    "size": 0.01,
})
```

### 5.3 不打印敏感信息

- ❌ API key、token、密码、UUID 完整值
- ✅ 仅打印前 4 位 + 后 4 位（`tt_xxxx...xxxx`）

## 6. 类型提示

### 6.1 全部公开 API 必须有 type hints

```python
def place_order(symbol: str, side: Side, qty: float) -> Order:
    ...
```

### 6.2 用 `dataclass` / `pydantic` 而非 `dict`

```python
# ❌ 散乱
def handle(payload: dict) -> dict:
    if payload['type'] == 'fill':
        ...

# ✅ 类型安全
@dataclass
class Fill:
    order_id: int
    price: float
    qty: float

def handle(fill: Fill) -> None:
    ...
```

## 7. 配置管理

### 7.1 不在代码里硬编码

- 所有可调参数放 `config/*.yml`
- 代码读 config 时用默认值兜底：`cfg.get('key', default)`
- 启动时打印实际生效的配置（敏感字段除外）

### 7.2 不入 git 的文件

放 `.gitignore`：
```
config/secrets.yml
data/
logs/
*.pyc
__pycache__/
.venv/
.env
```

## 8. 测试

### 8.1 单元测试

- 用 `pytest`
- 文件：`tests/unit/test_<module>.py`
- 函数：`test_<功能>_<场景>()`
- mock 外部依赖（交易所、DB）

### 8.2 集成测试

- 用真实 SQLite + paper 模式
- 端到端测试：策略 → 风控 → OMS → 模拟成交
- 文件：`tests/integration/test_<scenario>.py`

### 8.3 测试覆盖率（决策 B9，2026-05-01）

| 阶段 | 门槛 | CI gate |
|---|---|---|
| **Phase 1** | 每个新模块 ≥ 1 个 happy path + 1 个边界测试 | 不强制 coverage 数值 |
| **Phase 2 末** | 数据层 / 因子库 ≥ 70% | 软警告 |
| **Phase 3a 之前** | 风控 / OMS / 数据层 ≥ 80% | 强制 gate |
| **长期** | 策略层 ≥ 60%；整体 ≥ 70% | 强制 gate |

理由：起步阶段强 coverage gate 会拖慢节奏；先建好"测试习惯"，到 Phase 3a 涉及真钱前再硬卡。

## 9. Git 工作流

### 9.1 分支

- `main`：稳定可运行
- `develop`：开发主线（可选）
- `feature/<short-name>`：功能开发
- `fix/<short-name>`：bug 修复

### 9.2 提交信息

```
<type>: <短描述>

<可选详细说明>

<可选 footer：BREAKING CHANGE / Refs / Co-authored>
```

`<type>` 选：`feat / fix / docs / refactor / test / chore / perf`

例：
```
feat: 增加资金费率因子

实现 funding_rate 因子计算与 Parquet 落盘。
关联 ADR-X / Refs P2.1。
```

### 9.3 提交粒度

- 一次提交一件事
- 不混"添加功能"和"重构"
- 提交前本地跑一次测试

---

## 10. 时间和时区

- 内部统一 **UTC 毫秒时间戳**（int）
- 显示给人时再转本地时区（北京 UTC+8）
- 不存 ISO 字符串到 DB
- K 线 `ts` 字段是**开盘时间**

---

## 11. 数值精度

- 所有金额、价格、数量用 `float`（不用 Decimal，性能/便利权衡）
- 比较用 `abs(a - b) < epsilon` 而非 `a == b`
- 显示给人时按 symbols.tick_size 格式化

---

## 12. 模块大小

- 单文件 ≤ 500 行
- 单函数 ≤ 50 行
- 单类公开方法 ≤ 15 个
- 超过就拆

---

## 13. 待办标记

代码里的 TODO 必须包含：
- 谁
- 何时（年月）
- 为什么
- 链接 issue 或 ADR

```python
# TODO(blacker, 2026-Q3): 当 SQLite 超 10GB 时迁 PostgreSQL。
# 见 ADR-XXX。
```

避免 `# TODO: fix this later`。

---

## 14. 不可妥协的红线

以下规则**任何情况下都不允许违反**：

- 🚫 不允许跳过 paper 30 天直接上实盘
- 🚫 不允许任何形式的"未来函数"（用未来数据回测）
- 🚫 不允许测试集多次使用（OOS 必须只用一次）
- 🚫 不允许提交 secrets.yml 到 git
- 🚫 不允许在 critical 路径上吞异常
- 🚫 不允许直接改 prod DB（必须走 migration 脚本）
- 🚫 不允许杠杆 > 3x
- 🚫 不允许持仓没有交易所侧止损单
