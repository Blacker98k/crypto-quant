## 1. 整体存储策略

不同性质的数据用不同存储方案，避免单一存储成为瓶颈。

| 数据类型 | 存储 | 理由 |
|---|---|---|
| K 线（高频/海量） | **Parquet 分区文件** | 列式压缩、批量回测快、便宜 |
| 实时热数据 | **内存（dict + deque）** | 策略快速访问、不落库 |
| 交易事务（订单/成交/持仓） | **SQLite (WAL 模式)** | 事务安全、单机够用 |
| 信号/风险事件/日志 | **SQLite** | 关系查询、Dashboard 友好 |
| 因子值（中等量） | **DuckDB on Parquet** | 列式 + 即席查询 |
| 配置 | **YAML 文件** | 人可读、可热加载 |

> 当 SQLite > 10GB 或并发瓶颈时迁 PostgreSQL。当 Parquet > 100GB 时考虑 ClickHouse。

```
│   ├── BTCUSDT/

│   │   ├── 5m/

│   │       └── 2026.parquet         # ≥ 4h 按年
│   └── ETHUSDT/

│   └── BTCUSDT/
```

### 1.2 Parquet 分区粒度（B7 决策）

| 数据 | timeframe | 分区粒度 | 文件名 | 单文件预估行数 |
|---|---|---|---|---|
| K 线 | `1m` | 月 | `YYYY-MM.parquet` | ~43k |
| K 线 | `5m / 15m / 1h` | 月 | `YYYY-MM.parquet` | ~720 ~ 8.6k |
| K 线 | `4h / 1d` | 年 | `YYYY.parquet` | ~365 ~ 2200 |
| 资金费率 | 8h 周期 | 年 | `YYYY.parquet` | ~1095 |
| 持仓量 | 1m 拉一次 | 月 | `YYYY-MM.parquet` | ~43k |
| 订单簿快照 | 即时 | 日 | `YYYY-MM-DD.parquet` | 视情况 |
| 因子值 | 跟随原 K 线 | 同上 | 同上 | 同上 |

**原则**：单文件 1k-50k 行（DuckDB 列式扫描的最佳区间）。Phase 2 末加 compaction job 把"小且静态"的旧分区合并。

## 2. 数据真相源（Source of Truth）

> 系统里有 Parquet、SQLite、内存缓存、交易所 API 四类存储。**每类数据必须明确"谁是真相"**——服务重启 / 断线恢复 / Dashboard 展示时都信任真相源，副本可被重建。

### 2.1 真相源矩阵

| 数据类型 | 权威来源 | 副本/缓存 | 一致性策略 |
|---|---|---|---|
| 行情历史（K 线 / 资金费率 / OI） | **Parquet 文件** | DuckDB / 内存 | 单写多读，每周校验完整性 |
| 实时行情（最新价 / 订单簿 / 增量成交） | **交易所 WS 推送** | 内存（dict + deque） | 推送丢失即重订阅 |
| 订单 | **交易所 API + 本地 SQLite `orders`** | 内存映射 | **5min 主动对账**，差异告 critical |
| 成交回报 | **交易所 API + 本地 SQLite `fills`** | - | 推送 + 拉取双源校验 |
| 持仓 | **交易所 API（live）/ 本地 SQLite `positions`（paper）** | 内存映射 | live 模式以交易所为准，本地存副本；paper 模式以本地为准 |
| 策略信号 | **本地 SQLite `signals`** | 内存待处理队列 | 信号必须先落库才进入风控链路 |
| 风控事件 | **本地 SQLite `risk_events`** | - | 写优先；丢失视为系统故障 |
| 策略参数 / 系统配置 | **YAML 文件** | 内存 | 热加载；变更落 audit log |
| 因子值 | **Parquet 文件** | DuckDB | 单写多读 |
| 回测记录 | **本地 SQLite `backtest_runs`** | 报告 HTML 落盘 | 一次写入永不修改 |

### 2.2 关键约束（不可妥协）

1. **服务重启时**，所有内存状态 **必须能从权威来源完整重建**——任何"只在内存里"的状态都是 bug
2. **副本与权威不一致** 时，永远信任权威来源；不一致本身即风控事件，必须落 `risk_events`
3. **Dashboard 展示** 永远直接读权威源，**不展示任何"仅内存"** 的数据
4. **实盘订单与持仓**：本地 SQLite ≠ 交易所 → 触发 critical 告警，且**暂停所有新下单**直到对账完成
5. **Paper 模式**：本地 SQLite 即是真相源（无交易所对账）；**不允许**直接复用 live 数据库（避免污染）

### 2.3 一致性失败的处理

| 场景 | 检测方式 | 自动处置 |
|---|---|---|
| WS 行情推送中断 | 心跳超时 5s | 自动重订阅 + 用 REST 补漏 |
| 订单本地有 / 交易所无 | 5min 对账 | 标记本地订单 status=`reconcile_anomaly`，告警 |
| 订单交易所有 / 本地无 | 5min 对账 | 拉取交易所订单回填本地，告警 |
| 持仓数量不一致 | 5min 对账 | **暂停下单 + critical 告警**，等待人工确认后清算差额 |
| Parquet 文件损坏 | 启动时校验 + 抽样读 | 拒绝启动，要求从备份恢复 |

## 3. 实体清单

### 3.1 共享实体（研究 + 生产）

#### 3.1.1 symbols（标的字典）

```
symbols
├── id                INTEGER PK
├── exchange          TEXT      -- binance / okx
├── symbol            TEXT      -- BTC/USDT 或 BTCUSDT
├── type              TEXT      -- spot / perp / futures
├── base              TEXT      -- BTC
├── quote             TEXT      -- USDT

├── lot_size          REAL      -- 最小数量
├── min_notional      REAL      -- 最小订单价值
├── listed_at         INTEGER   -- 上线时间 ms
└── delisted_at       INTEGER   -- 下线时间 ms (NULL 表示活跃)
```

存 Parquet，逻辑 schema：

```
candles
├── symbol_id    INTEGER     -- 关联 symbols.id
├── ts           INTEGER     -- 开盘时间 ms (UTC)
├── o, h, l, c   REAL
├── v            REAL        -- 成交量 (base asset)
├── q            REAL        -- 成交额 (quote asset)

```

按 timeframe 分目录：`candles/{symbol}/{tf}/{yyyy-mm}.parquet`

```
funding_rates

```

```
open_interest
```

### 3.2 生产实体（仅生产用）

#### 3.2.1 signals（策略信号）

```
signals
├── id                  INTEGER PK
├── strategy            TEXT       -- 策略名 e.g. 's1_btc_trend'
├── strategy_version    TEXT       -- 策略代码 git short hash（溯源）
├── config_hash         TEXT       -- 当时策略参数的 SHA-256（溯源）
├── universe_version    TEXT       -- 当时使用的币池版本（每周一更新）
├── run_id              TEXT       -- 当前进程启动 UUID（同一进程内所有信号共享）
├── symbol_id           INTEGER    -- 标的
├── side                TEXT       -- long / short / close
├── entry_price         REAL       -- 建议入场价
├── stop_price          REAL       -- 止损价
├── target_price        REAL       -- 目标价
├── confidence          REAL       -- 0-1
├── suggested_size      REAL       -- 建议仓位（base asset）
├── time_in_force       TEXT       -- GTC / IOC / FOK / DAY
├── rationale           TEXT       -- JSON：决策依据
├── status              TEXT       -- pending / placed / rejected / skipped / expired
├── reject_reason       TEXT       -- 拒绝原因（如 'L3:exposure_cap'）
├── trade_group_id      TEXT       -- NULL=单腿；非空时关联 trade_groups.id（S3 套利等）
├── captured_at         INTEGER    -- 信号产生时间 ms
└── expires_at          INTEGER    -- 过期时间 ms（信号有时效）
```

**溯源字段为何冗余落到 signals**：将来复盘"这笔信号到底用的哪版策略 / 哪份参数 / 哪个币池"时，光靠 git 翻历史不够——参数可能在配置里被热改过；币池可能本周已经更新。这四个字段一起构成信号产生时的**完整上下文快照**，事后必能精确还原。

#### 3.2.2 orders（订单）

```
orders
├── id                  INTEGER PK
├── signal_id           INTEGER FK signals.id
├── client_order_id     TEXT       -- 幂等 key（自生成 UUID）
├── exchange_order_id   TEXT       -- 交易所返回的订单号
├── symbol_id           INTEGER FK
├── side                TEXT       -- buy / sell
├── type                TEXT       -- limit / market / stop / stop_limit / take_profit
├── price               REAL       -- 限价（市价单为 NULL）
├── stop_price          REAL       -- 触发价（仅止损/止盈）
├── quantity            REAL
├── filled_qty          REAL
├── avg_fill_price      REAL
├── status              TEXT       -- new / partial / filled / canceled / rejected / expired / reconcile_anomaly
├── parent_order_id     INTEGER    -- 关联的母单（止损单的母仓）
├── purpose             TEXT       -- entry / stop_loss / take_profit / exit / hedge_leg
├── time_in_force       TEXT
├── reduce_only         INTEGER    -- 0/1 (合约用)
├── trade_group_id      TEXT       -- NULL=单腿；非空时关联 trade_groups.id
├── strategy_version    TEXT       -- 复制自 signal（防 signal 被清理后追不到）
├── placed_at           INTEGER
└── updated_at          INTEGER
```

#### 3.2.4 positions（持仓）

```
positions
├── id                  INTEGER PK
├── symbol_id           INTEGER FK
├── strategy            TEXT       -- 哪个策略开的
├── strategy_version    TEXT       -- 复制自开仓 signal（溯源）
├── opening_signal_id   INTEGER    -- 关联首次开仓信号
├── side                TEXT       -- long / short
├── qty                 REAL
├── avg_entry_price     REAL
├── current_price       REAL       -- mark price
├── unrealized_pnl      REAL
├── realized_pnl        REAL
├── leverage            REAL       -- 合约用
├── margin              REAL       -- 占用保证金
├── liq_price           REAL       -- 强平价
├── stop_order_id       INTEGER    -- 关联交易所侧止损单
├── trade_group_id      TEXT       -- NULL=独立持仓；非空时是 trade_groups.id 的一腿
├── opened_at           INTEGER

```

#### 3.2.5 trade_groups（多腿事务组）

承载像 S3 资金费率套利这种**多腿原子交易**的元数据。一个 group 包含 ≥ 2 个 orders 和 ≥ 2 个 positions，用 `trade_group_id` 关联。

```
trade_groups
├── id              TEXT PK    -- UUID
├── strategy        TEXT       -- 'S3' 等
├── group_type      TEXT       -- 'spread_arb' (现货/永续套利) / 'pair_trade' / 'hedge'
├── status          TEXT       -- forming / hedged / unwinding / closed / broken / partial_alert
├── leg_plan        TEXT       -- JSON：计划的腿（symbol/side/qty/role）
├── leg_count       INTEGER    -- 计划腿数（套利通常 2）
├── filled_legs     INTEGER    -- 已对冲腿数
├── target_basis    REAL       -- 入场时记录的基差（套利用）
├── stop_basis      REAL       -- 基差扩大止损线
├── created_at      INTEGER
├── hedged_at       INTEGER    -- 双腿全部成交的时刻
├── closing_at      INTEGER    -- 开始平仓的时刻
└── closed_at       INTEGER    -- 全部平仓完成
```

**status 状态机**：

```
[forming] ── 第一腿成交 ──→ [hedging?] ── 第二腿在 30s 内成交 ──→ [hedged]
    │                          │
    │                          └── 第二腿 30s 未成交 ──→ [unwinding] ──→ [broken]
    │                                                                       (撤掉第一腿)
    │
    └── 第一腿失败 ──→ [broken]

[hedged] ── 出场触发 ──→ [unwinding] ── 双腿都平 ──→ [closed]
                          │
                          └── 仅一腿平掉 / 平仓失败 ──→ [partial_alert]
                                                       (人工介入)
```

## 4. 数据规范

### 4.1 时间

- **统一 UTC 毫秒时间戳**（INTEGER）
- 不存 ISO 字符串
- 显示时再转本地时区
- K 线 `ts` = 开盘时间，不是收盘时间

### 4.2 价格与数量

- 都用 `REAL`（双精度浮点）
- 显示用：根据 `symbols.tick_size` 决定保留位数
- 不要在 SQL 里做精度比较，用 `ABS(a-b) < tick_size` 而非 `a = b`

### 4.3 ID 生成

- 内部 ID：SQLite AUTOINCREMENT
- `client_order_id`：UUID v4，**幂等下单关键**
- 永远不复用 ID

### 4.4 状态机

- 状态字段都用 **TEXT** 枚举（不用整数）
- 状态变更必须更新 `updated_at`
- 只允许向前流转，不能回滚（特殊情况新增 `reverted` 状态）

### 4.5 软删除

- 不物理删除任何交易相关数据
- 通过 `status='canceled'` / `closed_at IS NOT NULL` 表示
- 历史可追溯是合规和复盘的基础

## 5. 关键约束与索引

### 4.1 唯一约束

| 表 | 唯一约束 | 用途 |
|---|---|---|
| symbols | (exchange, symbol, type) | 防重复 |
| orders | client_order_id | 幂等下单 |
| fills | (order_id, exchange_fill_id) | 防重复回报 |

### 4.2 主要索引

| 表 | 索引 | 用途 |
|---|---|---|
| signals | (status, captured_at) | 查未处理信号 |
| signals | (strategy, captured_at) | 按策略查信号 |
| orders | (status, placed_at) | 查活跃订单 |
| orders | (symbol_id, placed_at) | 按币查订单 |
| positions | (closed_at) WHERE closed_at IS NULL | 查活跃持仓 |
| risk_events | (captured_at, severity) | Dashboard 查警 |
| run_log | (captured_at) | 健康度时序 |

### 4.3 外键策略

- SQLite 默认不强制外键，**生产配置 `PRAGMA foreign_keys=ON`**
- 删除 symbols 不级联（保留历史）
- 删除 signals 不影响 orders（用 NULL 关联）

## 6. 迁移与版本

### 5.1 Schema 版本表

```
schema_version
├── version       INTEGER PK   -- 1, 2, 3...
├── applied_at    INTEGER
└── description   TEXT
```

### 5.2 迁移脚本规范

- 每次 schema 变更对应一份 SQL 文件：`migrations/V{n}__{desc}.sql`
- 脚本必须**幂等**（重复执行不出错）
- 启动时自动应用未执行的迁移
- 严禁手动改库（用脚本走流程）

### 5.3 向后兼容

- 加列：始终允许（默认值 NULL）
- 删列：禁止，标记 deprecated 即可
- 重命名：禁止（用新列 + 数据迁移）
- 改类型：仅允许扩大（INT → REAL 可以，反之禁止）
