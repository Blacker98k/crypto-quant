### 2026-05-09 · Dashboard 运行链路偏离核心设计

**现象**：Dashboard 长时间运行产生了大量 paper 记录，但记录来自探索策略和本地限速逻辑，不是 S1/S2/S3 与三层风控链路的结果。
**原因**：实时看板为了快速积累样本，先接了 dashboard 自有探索策略；核心策略和正式风控在 `core/` 中实现后，没有同步接回 dashboard runtime。
**解决**：Dashboard paper trader 接入核心策略 adapter 和正式风控 pipeline；默认策略切到 S1/S2，S3 在多腿执行支持前保持影子观察；补上回归测试。
**教训**：paper 证据必须来自与设计一致的运行链路，否则只能算探索样本，不能作为上线依据。
**关联**：`dashboard/paper_trading.py`、`core/risk/`、`core/strategy/`

### 2026-04-29 · LLM confidence 不可信（实测）

**现象**：在过往实测中，LLM（含主流推理模型）给出 0.60-0.65 的置信度，实际胜率仅 0.34-0.46，标定偏差 20-30 个百分点，且趋势稳定不是噪声。
**原因**：LLM 在金融场景普遍 over-confident，且没有针对该领域校准。
**解决**：不直接用 LLM confidence 作为开仓门槛，必须用历史数据做 calibration。
**教训**：本项目**禁止用 LLM 直接产生信号**。LLM 仅可用于**辅助打分**（如新闻情绪、事件归类），且必须以因子形式进入策略，再做标定。
**关联**：[ADR-007 禁止 LLM 直接产生信号](./07-决策日志/ADR-007-禁止LLM直接产生信号.md)

### 2026-04-29 · 入场质量是独立的 alpha 维度

**现象**：在过往离散下单类项目中观察到：方向预测胜率仅约 41%，但**锁价时点比市场最终定价更优**（即"入场比市场便宜"）的样本，ROI 能录得 +9%。
**原因**：好的入场价格是 alpha 的来源之一，与方向判断是独立维度。
**解决**：本项目对应概念为**滑点监控**——下单后追踪实际成交价与预期价偏差，并把它作为评估策略的独立维度。
**教训**：策略评估不能只看胜率，要拆出**入场质量** vs **方向判断**两个独立指标。

### 2026-04-29 · 跳过回测直接 live 必死

**现象**：过往项目曾完全跳过回测、直接小资金实盘，结果短期内出现统计上无显著性但走势已暴露问题的负收益，模型 calibration 在样本量足够前就已经亏掉了部分本金。
**原因**：没有事前评估机制就是在用真钱做随机实验，币圈杠杆会把这种损失放大 10 倍。
**解决**：本项目 **Phase 1 必须先做回测引擎**，Phase 3 之后才上实盘。
**教训**：硬性规则——**任何策略实盘前 paper 跑 ≥ 30 天 + OOS 验证通过**。
**关联**：[ADR-002 Phase 1 先做回测](./07-决策日志/ADR-002-Phase1先做回测.md)

### 2026-04-29 · token / session 失效不监控会静默停摆

**现象**：过往项目中遇到过：第三方平台 token 寿命 36-48 小时，到期后程序持续报"网络错误"，实际是 unauthorized。本地报错信息误导，靠人眼一直发现不了。
**原因**：HTTP 客户端连接池里有死连接，导致请求未真正发出，错误显示为 RequestError 而非 unauthorized。
**解决**：定期重启 + 监控 unauthorized 计数 + 面板"凭据健康"判定加"unauth 后是否有 ok"二阶判断。
**教训**：本项目用交易所 API key（不会过期），但 **rate limit / IP ban / 签名失效等异常必须有明确的健康监控**。

### 2026-04-29 · 长连接 / HTTP 连接池长时间运行会腐化

**现象**：过往项目程序运行 36 小时后所有 httpx 请求失败，重启就好。
**原因**：HTTP/HTTPS 长连接池有死连接，SDK 没自动剔除。
**解决**：定期重启 / 设置连接最大寿命 / 用每次新建 client（性能损失可接受）。
**教训**：长期运行的服务必须有"健康自检 + 自动恢复"机制。

### 2026-04-29 · 全局代理变量污染本地访问

**现象**：环境里设了 `http_proxy=127.0.0.1:57777`，访问 `127.0.0.1:8088` Dashboard 也走代理，返回 502。
**原因**：`no_proxy` 没正确设置，curl/python 走代理访问本地。
**解决**：脚本启动时显式 `unset http_proxy`（仅在访问本地服务前），或在 `no_proxy` 加全 IP 段。
**教训**：本项目脚本启动时**显式设置 `no_proxy=localhost,127.0.0.1,0.0.0.0`**。

### 2026-05-02 · Cursor Shell 给 git 加 --trailer 老版本不认

**现象**：在 Cursor Shell 里执行 `git commit` 报 `error: unknown option 'trailer'`；本机 Git 为 2.20.1，不支持 `--parse` / trailer 相关语法。
**原因**：IDE 外层对 `git commit` 注入了 trailer 相关附加参数，与老版本 Git CLI 不兼容。
**解决**：绕开包装层，直调系统 Git 并显式传入作者（本机未配 `user.name` 时一并带上）：
`/usr/bin/git -c user.name="…" -c user.email="…" commit -F COMMIT_MSG_FILE`
**教训**：Agent 自动化提交前先看 `git --version`；< 2.22 环境避免依赖 trailer / 新子命令；或用 `git commit` 文件模式减少转义问题。
**关联**：PR #3 提交过程

### 2026-05-02 · gh 凭据助手子进程拿不到 HTTPS_PROXY

**现象**：`git push` 卡住几十秒无输出，Ctrl+C 后才能继续排查；加 `HTTPS_PROXY`/`HTTP_PROXY` 环境变量仍无效。
**原因**：`gh auth git-credential` 子进程未继承代理；Git 侧也未设置 `http.proxy` / `https.proxy`。
**解决**：push 时同时设环境变量 **和** Git 配置：
`git -c http.proxy=http://127.0.0.1:57777 -c https.proxy=http://127.0.0.1:57777 push …`
或在 `~/.gitconfig` `[http]` 永久写 `proxy`。
**教训**：国内推 GitHub：**环境变量 alone 不够**，要给 Git 本体也配 proxy。
**关联**：PR #3 push

### 2026-05-04 · ruff RUF023 __slots__ 必须自然序排列

**现象**：`ruff check` 报 `RUF023 __slots__ is not sorted`，要求按字母序排列所有 slot 名称。
**原因**：ruff 的 RUF023 规则强制 `__slots__` 成员按字母序排列，与代码中逻辑分组的直觉冲突。
**解决**：将 `__slots__` 按字母序重排（如 `_cache, _engine, _last_price, _last_ts, _paper_trading, _parquet_io, _repo, _running, _thread`）。
**教训**：dataclass/slots 相关文件写完就跑 `ruff check --fix` 自动排序；人工排容易漏。

### 2026-05-04 · E402 — sys.path 插入导致 import 不在文件顶部

**现象**：`dashboard/server.py` 需要在导入 `core.*` 之前把项目根注入 `sys.path`，导致所有后续 import 触发 `E402 module level import not at top of file`。
**原因**：Python 的 import 必须在文件顶部的规则与"先配置 sys.path 再 import 本地包"的实际需求冲突。
**解决**：在受影响的 import 行尾加 `# noqa: E402` 注释，共 9 行。这是合理的例外——`sys.path` 注入必须在本地包导入之前。
**教训**：需要 `sys.path` hack 的入口脚本，在文件头注释清楚原因，批量加 noqa 即可，不要为此拆 `__main__` 包装文件。

### 2026-05-04 · RUF012 — 类级别可变默认值需 ClassVar

**现象**：`DataSimulator` 的 `SYMBOLS` / `TIMEFRAMES` 类属性被 ruff 报 `RUF012 Mutable default value for class attribute`。
**原因**：Python 的 list 默认值是可变对象，所有实例共享同一引用；虽然这里只是常量不会被修改，但 ruff 无法推断这一点。
**解决**：加 `from typing import ClassVar`，注解为 `SYMBOLS: ClassVar[list[str]]`。语义上明确"这是类级常量，不会被实例修改"。
**教训**：类级 list/dict 常量加 `ClassVar` 注解，既满足 lint 又表明意图。

### 2026-05-04 · PaperMatchingEngine 的 FOREIGN KEY 约束

**现象**：在测试中调用 `engine.place_order(intent)` 时，SQLite 报 `FOREIGN KEY constraint failed`，因为 `orders.symbol_id` 引用 `symbols` 表但表中无数据。
**原因**：引擎的 `_resolve_symbol_id()` 查询 `symbols` 表查找 symbol，如果找不到就尝试插入。但测试环境中的临时数据库没有预置 symbol 数据。此外 `orders.signal_id` 也引用 `signals` 表。
**解决**：在 paper engine 测试中加 `autouse=True` fixture `_stub_symbol_and_signal`，预置 `BTCUSDT`/`ETHUSDT` 到 `symbols` 表，并插入一条桩 `signals` 行。
**教训**：Repository 模式下的测试要确认 FK 依赖链——`orders` 依赖 `symbols` 和 `signals`，缺一不可。推荐写 autouse fixture 而非在每个测试里手动插入。

### 2026-05-04 · DataSimulator 线程与 FastAPI asyncio 的隔离

**现象**：DataSimulator 用 `threading.Thread` 跑死循环写 MemoryCache + SQLite，FastAPI 用 asyncio 事件循环处理 HTTP 请求读同一份数据。
**原因**：两个并发模型（线程 vs asyncio）共享可变状态，不加保护会有竞态。
**解决**：MemoryCache 自带 `threading.Lock`，put/get 线程安全；SQLite 用 WAL 模式 + `check_same_thread=False` 允许跨线程读写；DataSimulator 只在 bg 线程写，API 只在 asyncio 协程读，方向单一降低冲突。
**教训**：线程与 asyncio 混合架构中，关键是**数据桥**的线程安全。MemoryCache 加锁 + SQLite WAL 是轻量方案。如果未来写入方变多（多个 simulator 线程），需要升级到生产者-消费者队列。

### 2026-05-04 · ccxt timeout 设成秒而非毫秒 → 所有请求都超时

**现象**：`BinanceSpotAdapter` 一直报 `_ssl.c:999: The handshake operation timed out`，但 curl 和直接 `requests.get` 正常。

**原因**：`BinanceSpotAdapter.__init__` 里把 `ccxt_config["timeout"]` 写成 `timeout_ms / 1000.0`，但 ccxt 官方文档定义的 timeout 单位是**毫秒**。默认 `timeout_ms=5000` 经过除法变成了 `5.0ms` 发给 ccxt，导致 5ms 内连 SSL 握手都来不及完成。

**解决**：直接传 `"timeout": timeout_ms`，去掉 `/ 1000.0`。

**教训**：ccxt 4.x 的 timeout 单位是毫秒（与 config 字段名 `timeout_ms` 一致），新版本的迁移指南可能变了，但旧代码没跟上。初次接入 ccxt 时写个连通性单测可提前发现这类问题。
