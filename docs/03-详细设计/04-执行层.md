# 04 · 执行层（OMS / 强制止损 / 对账）

> **摘要**：信号通过三层风控后落到这一层。本模块只做"提交订单 → 跟踪状态 → 强制保护持仓 → 与交易所对账"。**不做风控判断**。
> **关联**：[架构 §5 链路](../02-架构设计文档.md#5-信号-决策-执行链路) · [ADR-003 强制止损](../07-决策日志/ADR-003-强制交易所侧止损.md) · [ADR-008 三层风控冗余](../07-决策日志/ADR-008-三层风控冗余.md) · [数据模型 §3.2 生产实体](../05-数据模型.md#32-生产实体仅生产用)

## 1. 模块边界

### 输入

来自 L1 风控通过的"已校准下单意图"（不是原始 Signal）：

```
DownstreamOrderIntent
├── signal_id           -- 来源信号
├── trade_group_id      -- 多腿组（NULL=单腿）
├── symbol
├── side                -- buy / sell
├── type                -- limit / market / stop / take_profit
├── price               -- 限价（市价为 NULL）
├── quantity            -- 已经过 L2 sizer
├── tif                 -- GTC / IOC / FOK
├── reduce_only         -- 0/1
├── stop_loss_price     -- 强制止损价（**非空时 OMS 必须挂止损单保护**）
└── take_profit_price   -- 可选目标价
```

### 输出

- **已落库的订单**：写 `orders` 表
- **成交回报**：写 `fills` 表 + 更新 `positions`
- **强制止损单**：每个开仓订单成交后必有对应止损（`orders.purpose = 'stop_loss'`）
- **风险事件**：止损失败 / 对账异常 / API 限流等都写 `risk_events`

### 依赖

- `ExchangeAdapter`（封装 ccxt-pro，含 REST + WS）
- `Database`（SQLite）
- `Clock`（时间源，便于回测注入）
- 不依赖：`Strategy` / `RiskEngine`（执行层在它们下游）

## 3. OrderStateMachine（订单状态机）

```
                  ┌──────────┐
                  │ NEW      │ (本地生成 client_order_id)
                  └────┬─────┘
                       │ submit()
                       ▼
                  ┌──────────────┐
                  │ PENDING_NEW  │ (已发出但未收到 exchange_order_id)
                  └────┬─────────┘
              ack ─────┤  reject
                       ▼          ──────────┐
                  ┌──────────┐               │
                  │ ACCEPTED │               ▼
                  └────┬─────┘          ┌──────────┐
            partial ───┤    fill        │ REJECTED │
                       ▼                └──────────┘
              ┌──────────────────┐
              │ PARTIALLY_FILLED │
              └────┬─────────────┘
              fill │   timeout
                   ▼
              ┌──────────┐                 ┌──────────┐
              │ FILLED   │                 │ CANCELED │
              └──────────┘                 └──────────┘
                                            ┌──────────┐
                                            │ EXPIRED  │
                                            └──────────┘
```

**状态规则**：

- 状态只向前流转，不允许回退
- 每次状态变更必须更新 `orders.updated_at`
- 进 `PARTIALLY_FILLED` 后，5 分钟内未达 `FILLED` 则触发"追单"（改价或撤单）
- `REJECTED` / `CANCELED` / `EXPIRED` 是终态，不再处理

## 4. 强制止损状态机（核心）

> 这是本项目最关键的保护机制。详细背景见 [ADR-003](../07-决策日志/ADR-003-强制交易所侧止损.md)。

### 4.1 核心契约

**任何带 `stop_loss_price` 的 entry 订单成交后，OMS 必须在 5 秒内成功挂出对应的交易所侧止损单**。否则 → 自动平仓该持仓。

### 4.2 状态机

```
[ENTRY_FILLED] (开仓成交，部分或全部)
      │
      │ 立即（同事务内）
      ▼
[STOP_PENDING] (尝试挂止损)
      │
      ├─── 挂单成功 ─→ [STOP_PROTECTED] ✅ 正常持仓状态
      │
      ├─── 5s 内未成功 ─→ [STOP_TIMEOUT] ─→ 触发 [AUTO_CLOSE]
      │
      ├─── 交易所 reject ─→ [STOP_REJECTED] ─→ 按拒绝原因分支
      │                                       (见 §4.4)
      │
      └─── 网络断 / API 限流 ─→ [STOP_RETRYING] ─→ 重试 3 次
                                                   ├─ 成功 → STOP_PROTECTED
                                                   └─ 失败 → STOP_UNPROTECTED → AUTO_CLOSE

派生状态（用于 §4.4 失败分支判断）：
- [STOP_PROTECTED]   — 止损已确认在交易所侧；最近一次确认时间 < 60s
- [STOP_UNKNOWN]     — 止损可能在也可能不在（最近一次确认时间 ≥ 60s）；遇网络断时按风险更高场景处理
- [STOP_UNPROTECTED] — 已知止损未挂上 / 已被取消 / 已被触发但持仓未平

[STOP_PROTECTED]
      │
      ├─── 持仓变更（加仓/部分平仓） ─→ [STOP_AMENDING] ─→ 撤旧+重挂
      │
      ├─── 止损被触发成交 ─→ [STOP_TRIGGERED] ─→ 持仓平掉，记录 fills
      │
      └─── 持仓全部平掉 ─→ [STOP_CANCELED] (手动撤掉残留止损)

[AUTO_CLOSE] = 市价平掉持仓 + critical 告警 + 落 risk_events
```

### 4.3 关键时间约束

| 约束 | 阈值 | 失败动作 |
|---|---|---|
| 开仓→挂止损 | ≤ 5s | AUTO_CLOSE |
| 单次挂止损 API 重试 | ≤ 3 次（指数退避 200ms / 500ms / 1.5s） | AUTO_CLOSE |
| 止损单状态查询 | 每 30s 一次 | 不一致 → 重挂 |
| 持仓变更后改止损 | ≤ 10s | AUTO_CLOSE |

### 4.4 9 类失败分支处置矩阵

每一类都必须有自动化测试覆盖：

| # | 失败场景 | 检测方式 | 自动处置 | 监控指标 |
|---|---|---|---|---|
| 1 | **开仓部分成交**：`filled_qty < quantity` | OrderStateMachine fill event | 止损单按已成交量挂；未成交剩余继续等或撤掉 | `partial_entry_count` |
| 2 | **止损单被交易所拒**：reason=`PRICE_FILTER` 等 | 提交后收到 reject | 校正价格到 tick_size 倍数后重挂 1 次；仍失败 → AUTO_CLOSE | `stop_reject_count` by reason |
| 3 | **挂止损 5s 超时**：未收到 ack | OMS 计时器 | AUTO_CLOSE + critical 告警 | `stop_timeout_count` |
| 4 | **撤旧止损失败**（持仓变更要换止损时） | exchange API 返回 error | 重试 3 次；都失败 → 不撤旧的，先挂新的（交易所允许重叠则 OK；否则 AUTO_CLOSE） | `stop_amend_failure` |
| 5 | **重挂止损失败**（撤旧成功但新止损挂不上） | exchange API error | 等同 STOP_TIMEOUT → AUTO_CLOSE | `stop_remit_failure` |
| 6 | **API 限流**（HTTP 429 / Binance -1003） | exchange adapter 识别 | 退避重试（按交易所 retry-after），3 次后 AUTO_CLOSE；同时全局降速 | `rate_limit_hit` |
| 7a | **网络断开 + 持仓已是 `STOP_PROTECTED`**（止损单确认在交易所侧已挂上） | 健康检查 + 止损状态最近一次确认时间 < 60s | 进入 `degraded` 模式：拒绝新开仓；交易所侧止损独立运转，本地等待恢复；恢复后第一件事对账止损单状态 | `network_down_seconds` `stop_protected_during_outage` |
| 7b | **网络断开 + 持仓为 `STOP_UNKNOWN` / `STOP_UNPROTECTED`**（止损未挂上 / 状态未确认 / 最近一次确认 ≥ 60s 前） | 同上 + 持仓状态查询 | **critical 告警 + 全局拒绝新开仓 + 启动备用网络通道**（如 4G 备线）；备用通道仍不通 → 维护者必须人工接管，可能直接登录交易所网页 / app 手动平仓 | `unprotected_during_outage`（任何 > 0 都是严重事件） |
| 8 | **`reduce_only` 失败**（合约模式下挂 reduce-only 止损被拒） | reject reason 含 `REDUCE_ONLY_REJECTED` | 改为非 reduce_only 重发；监控是否反向开仓（若交易所行为异常，立即 AUTO_CLOSE） | `reduce_only_reject` |
| 9 | **持仓模式不一致**（hedge mode vs one-way mode） | 启动时探测 | 与配置不符 → **拒绝启动 OMS**（不允许在错误的持仓模式下下单） | `position_mode_mismatch` |

### 4.5 AUTO_CLOSE 实现

```
AUTO_CLOSE(position_id, reason):
    1. 在本地标记持仓 status = 'auto_closing'
    2. 立即发市价平仓单（reduce_only=true，IOC）
    3. 落 risk_events: severity=critical, type='stop_loss_failed', reason=...
    4. Telegram 推送 critical 告警
    5. 平仓订单状态机正常处理；若市价平仓也失败：
       - 全局熔断（拒绝所有新信号）
       - 持续重试市价平仓
       - 维护者必须人工介入
```

## 5. TradeGroupCoordinator（双腿原子化）

仅 S3 资金费率套利等多腿策略用。详见 [`数据模型 §3.2.5 trade_groups`](../05-数据模型.md#325-trade_groups多腿事务组) 状态机。

OMS 在收到带 `trade_group_id` 的订单时：

1. 第一腿先发单
2. 第一腿成交后，30 秒内必须完成第二腿
3. 否则 `unwinding`：撤第二腿 + 市价反向平第一腿
4. 双腿都成交后，group `status = hedged`，进入持仓周期
5. 出场（人工触发或风控触发）必须双腿同时撤；任一失败 → `partial_alert` + 暂停该策略新信号

详细失败矩阵见 [S3 spec §6.2](./strategies/S3-资金费率套利.md#62-双腿失败矩阵)。

## 6. Reconciler（对账）

每 5 分钟自动对账一次，覆盖：

| 项 | 本地源 | 交易所源 | 差异处置 |
|---|---|---|---|
| 活跃订单 | `orders WHERE status IN (new, partial)` | `GET /openOrders` | 本地有交易所无 → 标 `reconcile_anomaly` + 告警 |
| 持仓 | `positions WHERE closed_at IS NULL` | `GET /positionRisk` | 数量不一致 → critical + **暂停下单** |
| 余额 | `balance_snapshots` 最新 | `GET /balance` | 偏差 > 1% → 告警 |
| 止损单 | `orders WHERE purpose='stop_loss' AND status='accepted'` | 同活跃订单查询 | 止损不存在 → 立即重挂 |

对账失败本身就是风控事件。**对账 3 次连续失败 → 进入 `degraded` 模式拒绝新开仓**。

## 7. 异常与降级

| 模式 | 触发 | 行为 |
|---|---|---|
| `normal` | 默认 | 全功能 |
| `degraded` | 网络异常 / 对账连续失败 | 拒绝新开仓；现有持仓继续监控；尝试恢复 |
| `frozen` | 风控熔断 / AUTO_CLOSE 多发 | 拒绝所有动作；维护者必须人工解冻 |
| `paper` | 配置 `mode: paper` | 调用 paper 撮合器，不发真实订单 |

降级是**主动**进入：检测到异常立即降级，恢复需要明确条件而不是自动恢复。

## 8. 性能考量

- 单笔订单从生成到提交：< 200ms（不含网络往返）
- 强制止损挂单延迟：< 1s（从开仓 fill 起算）
- 5min 对账：< 30s 完成
- 单进程支持 ≥ 50 个并发活跃订单

瓶颈预期：
- WS 推送处理（含解码）
- SQLite 写入（用 WAL 模式 + 批量提交缓解）

## 9. 测试策略

| 层 | 类型 | 关键案例 |
|---|---|---|
| 单测 | OrderStateMachine | 状态转换合法性 / 非法转换被拒 |
| 单测 | StopLossManager | 9 类失败分支每个都有 case |
| 集成 | OMS + 模拟 ExchangeAdapter | 完整 entry → stop → fill → reconcile 链路 |
| 集成 | TradeGroupCoordinator | 双腿成交 / 半腿超时 / 平仓失败 |
| 演练 | 故障注入 | 模拟网络断 / 限流 / reject，验证 AUTO_CLOSE |
| 实盘 | paper 模式 | 30 天连续运行无止损失误 |
