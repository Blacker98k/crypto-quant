## 1. 模块边界

### 输入

| 来源 | 内容 | 频率 |
|---|---|---|
| Binance Spot REST `/api/v3/klines` | 历史 K 线 | 一次性回填 + 增量 |
| Binance USDM REST `/fapi/v1/klines` | 永续历史 K 线 | 同上 |
| Binance Spot WS `<symbol>@kline_<tf>` | 实时 K 线 | 推送 |
| Binance USDM WS `<symbol>@kline_<tf>` `<symbol>@markPrice` `<symbol>!fundingRate@arr` | 实时 K 线 / 标记价 / 资金费率 | 推送 |
| Binance REST `/fapi/v1/openInterest` | 持仓量（按需） | 1m 拉一次 |

外部接口的请求 / 错误码细节统一在 [`接口/04-外部接口.md`](../04-接口文档/04-外部接口.md)，本文不重复。

### 输出

| 输出 | 落点 | 真相源 |
|---|---|---|
| K 线（已收盘） | `data/candles/{symbol}/{tf}/{partition}.parquet` | Parquet |
| 资金费率 | `data/funding/{symbol}/{partition}.parquet` | Parquet |
| 持仓量 | `data/oi/{symbol}/{partition}.parquet` | Parquet |
| 标的字典 | SQLite `symbols` 表 | SQLite |
| 接口健康 | SQLite `run_log` 表 | SQLite |
| 实时最新价 / 最近 N 根 K 线 | 进程内存（dict + deque） | WS 推送（可重建） |

真相源矩阵详见 [`数据模型 §2`](../05-数据模型.md#2-数据真相源source-of-truth)。

## 2. 内部结构

```
core/data/
├── __init__.py
├── feed.py              # DataFeed Protocol + LiveFeed / ResearchFeed 实现
├── memory_cache.py      # 实时热数据：dict + deque
├── parquet_io.py        # 写 / 读 / 校验 / 分区路由
├── sqlite_repo.py       # symbols / run_log / kv 等 SQLite 操作
├── backfill.py          # BackfillJob：历史回填 + 增量
├── ws_subscriber.py     # WS 订阅 + 断线重连 + 心跳
├── gap_detector.py      # K 线缺洞检测 + REST 补漏
├── universe.py          # 币池筛选（核心 / 观察 / 普查）
├── exchange/
│   ├── base.py          # ExchangeAdapter Protocol
│   ├── binance_spot.py
│   └── binance_usdm.py
└── symbol.py            # Symbol 命名规范化（BTCUSDT 风格）
```

### 关键组件职责

| 组件 | 职责 | 状态 |
|---|---|---|
| `Symbol` | 字符串规范化（`BTC/USDT` ↔ `BTCUSDT` ↔ `(exchange,symbol,type)`） | 纯函数 |
| `MemoryCache` | 维护每 (symbol, tf) 一条 deque（默认保留 1000 根） + 最新价 dict | 线程安全 |
| `ParquetIO` | 写：append + 按分区合并；读：通过 DuckDB；启动时校验文件完整性 | 单写多读 |
| `BackfillJob` | 拉历史，按 symbol+tf 切片并发，写 parquet | 一次性 + 可断点续 |
| `WsSubscriber` | 长连接订阅，心跳超时 5s 自动重连，断线期间 REST 补漏 | 守护线程 |
| `GapDetector` | 启动 / 周期触发：扫 parquet 找缺洞，调 REST 补 | 周期 |
| `Universe` | 按 24h 成交额筛 top N，分层落 `symbols.universe` 字段 | 周期 |
| `LiveFeed` | 生产 DataFeed：拉取走 SQLite/Parquet/内存，订阅走 WS | 单例 |
| `ResearchFeed` | 研究 DataFeed：拉取走 Parquet，订阅 = 顺序回放 | 按需 |

## 3. 控制流

### 3.1 冷启动（首次 / 长时间停机后）

```
1. SqliteRepo.run_migrations()              -- 应用未执行的 V*__*.sql
2. ExchangeAdapter.fetch_exchange_info()    -- 拉 symbol meta（tick_size 等）
   → 落 symbols 表
3. Universe.rebuild('core')                 -- 算 top 20 + 落 symbols.universe
   Universe.rebuild('observation')          -- 21-100
   Universe.rebuild('scan')                 -- 101-300
4. BackfillJob.run(scope=all_universe)      -- 按需补足 N 年历史
   并发 ≤ 5（避开 REST rate limit）
   每个 (symbol, tf) 进度落 SQLite kv（断点续）
5. GapDetector.scan_all()                   -- 校验回填后无缺洞
6. WsSubscriber.start(scope=core+observation)
7. 通知就绪：写 run_log(endpoint='startup', status='ok')
```

### 3.2 热启动（崩溃恢复 / 重启）

```
1. SqliteRepo.run_migrations()
2. 读 SQLite kv 取上次 BackfillJob 进度
3. BackfillJob.run(scope=resumable)         -- 仅补未完成的 (symbol, tf)
4. GapDetector.scan_all()
5. WsSubscriber.start(scope=last_active)    -- 从 SQLite 恢复上次的订阅集
```

**约束**：内存热数据（最近 N 根 K 线）启动时必须从 Parquet **完整重建**——任何"只在内存里"的历史都是 bug（[数据模型 §2.2](../05-数据模型.md#22-关键约束不可妥协)）。

### 3.3 实时 K 线推送链路

```
WS 推送 (kline event, closed=true|false)
   │
   ▼
ws_subscriber.on_message()
   │  ├── decode → Bar 数据类
   │  ├── 写入 MemoryCache（覆盖未收盘 / append 已收盘）
   │  └── if closed: 投递到 ParquetWriter 异步队列 + 通知订阅方
   ▼
ParquetWriter.flush_loop()
   │  ├── 按 (symbol, tf, partition) 攒批，每 60s 或 每 1000 条 flush
   │  └── 写入失败 → 重试 3 次 → 仍失败 → critical 告警 + 落 run_log
   ▼
DataFeed 订阅者收到 Bar（closed=True）
```

**Backpressure**：MemoryCache → 订阅者 callback 用有界队列（默认 1000）。满了丢最旧 + 落 run_log（[数据接口 §5.3](../04-接口文档/02-数据接口.md#53-backpressure)）。

### 3.4 历史拉取链路（research / live 共用）

```
DataFeed.get_candles(symbol, tf, start_ms, end_ms)
   │
   ├── (live 模式) 先查 MemoryCache 看请求区间的尾部是否在内存
   │     ├── 是 → 内存命中部分 + Parquet 读剩余 → 合并按 ts 排序
   │     └── 否 → 完全走 Parquet
   │
   └── (research 模式) 永远走 Parquet/DuckDB
       └── DuckDB SQL：SELECT * FROM read_parquet(...) WHERE ts BETWEEN ?
```

**幂等性**：同参数返回相同结果（[数据接口 §7.1](../04-接口文档/02-数据接口.md#71-幂等性)）。

## 4. 关键算法

### 4.1 K 线缺洞检测

每个 timeframe 对应一个标准间隔（1m=60_000ms, 4h=14_400_000ms 等）。扫描算法：

```
gaps = []
prev_ts = None
for bar in sorted(parquet_rows, by=ts):
    if prev_ts is not None and bar.ts - prev_ts > tf_interval_ms:
        gaps.append((prev_ts + tf_interval_ms, bar.ts))
    prev_ts = bar.ts
return gaps
```

补洞动作：调 REST `/klines?startTime&endTime` 拉缺失区间，**追加写入 parquet 后立即重新校验**。

**容忍度**：交易所偶有维护停盘（如 Binance 每月一次小时级停盘），缺洞 ≤ 1h 自动忽略 + 落 `run_log`；> 1h 触发告警，由维护者人工确认是否补。

### 4.2 Parquet 分区路由

| timeframe | 分区粒度 | 文件名格式 | 单文件预估行数 |
|---|---|---|---|
| `1m` | 月 | `2026-01.parquet` | ~43k |
| `5m` | 月 | `2026-01.parquet` | ~8.6k |
| `15m` | 月 | `2026-01.parquet` | ~2.9k |
| `1h` | 月 | `2026-01.parquet` | ~720 |
| `4h` | 年 | `2026.parquet` | ~2.2k |
| `1d` | 年 | `2026.parquet` | ~365 |
| 资金费率 | 年 | `2026.parquet` | ~1095（每 8h） |

> 这条规则替代了 [数据模型 §1.1](../05-数据模型.md#11-物理布局) 里"统一按月"的早期描述。≤ 1h 用月分区；≥ 4h 用年分区，避免文件过碎。

### 4.3 命名规范化（Symbol）

对内统一无分隔符大写（`BTCUSDT`），对外展示按需转换。`Symbol` 工具函数：

| 输入 | 输出（normalize） |
|---|---|
| `BTC/USDT` | `BTCUSDT` |
| `btcusdt` | `BTCUSDT` |
| `BTC-USDT-SWAP` | `BTCUSDT`（type=`perp`） |

**消歧靠 `(exchange, symbol, type)` 三元组**——同名的现货与永续在 SQLite 用复合唯一约束区分。Symbol 自带 type 后缀的（如 OKX 的 `-SWAP`）解析到 `type` 字段后丢掉后缀。

### 4.4 币池筛选

```
score(symbol) = 24h_quote_volume(symbol)  # 用 USDT 计价的成交额
```

按规则集排序：

1. 拉 `GET /api/v3/ticker/24hr`（spot）+ `GET /fapi/v1/ticker/24hr`（perp）
2. 过滤：稳定币（USDC/DAI/TUSD/FDUSD/...）、绑定币（BNB/...）、上线 < 30 天
3. 按 `quote_volume` 降序取 top 20 / 21-100 / 101-300，分别落 `universe='core'/'observation'/'scan'`
4. 旧记录的 `universe` 字段 update 为新值；从池子掉出去的标记为 `NULL`（不删行）
5. 写一行到 `universes` 元数据表（`version=YYYYMMDD-HHMM`），供 `signals.universe_version` 引用

**频率**：core / observation 每周一 03:00 UTC；scan 每月 1 号。手动触发支持 `python scripts/rebuild_universe.py --layer core`。

> **Phase 1 简化**：`universes` 元数据表先不上，`universe_version` 字段用硬编码 `'p1-stub'` 占位。Phase 2 末（研究区完善）再补完整版本。

## 5. 异常与降级

### 5.1 WS 异常

| 场景 | 检测 | 处置 |
|---|---|---|
| 心跳超时 5s | ping/pong 计时器 | 立即重订阅；恢复后立即调 REST 补这段缺失的 closed bar |
| 单 stream 持续报错 | 同一 stream 连续 3 次重连失败 | 隔离该 stream（不影响其他），critical 告警 |
| 全局 WS 不通 ≥ 60s | 所有订阅心跳同时超时 | 进入 `data degraded` 模式：DataFeed.subscribe_* 暂停推送；策略依赖 WS 的 on_bar 不被调用；Dashboard 显示红色 |

### 5.2 REST 异常

| 场景 | 处置 |
|---|---|
| HTTP 429 / Binance -1003（rate limit） | 按响应头 `Retry-After` 退避；同时**全局 REST 限流器**降速 |
| HTTP 5xx | 指数退避重试 3 次（200ms / 500ms / 1.5s） |
| HTTP 4xx（非 429） | 不重试，直接 raise + 落 run_log（编程错误或 symbol 不存在） |
| 网络超时 | 与 5xx 同 |

详细错误码到异常类的映射见 [`接口/04-外部接口.md §4`](../04-接口文档/04-外部接口.md)。

### 5.3 Parquet 文件损坏

启动时 `ParquetIO.verify_all()` 抽样读每个分区文件 1000 行：

- 读失败 → 标记 `corrupt`，**拒绝启动数据层**（数据真相源不可信）
- 维护者必须从备份恢复或重新回填后才能继续

### 5.4 数据洞容忍

| 缺洞规模 | 自动处置 | 告警 |
|---|---|---|
| ≤ 1h | 自动 REST 补 | 落 `run_log`（info） |
| 1h < gap ≤ 24h | 自动 REST 补 | 告警（warn） |
| > 24h | 不自动补，等维护者决策 | critical |

## 6. 性能考量

### 6.1 预算

| 项 | 目标 |
|---|---|
| WS 推送解码 → 推到订阅者 | < 50ms |
| `get_last_price()` | < 1ms（内存命中） |
| `get_candles(n=200)` | < 50ms（内存 + parquet 混合） |
| `get_candles(start, end)` 跨年度 | < 500ms（DuckDB 列式扫描） |
| Parquet 写延迟 | 异步队列，永不阻塞 WS callback |
| 单进程同时订阅数 | ≥ 200 个 (symbol, tf) 流 |

### 6.2 瓶颈预测

| 瓶颈 | 缓解 |
|---|---|
| WS 单连接吞吐 | Binance 单连接限 200 stream；分多连接（spot/perp 至少分开） |
| SQLite 写并发 | WAL 模式 + 写动作集中到 `sqlite_repo` 单线程，其他线程入队 |
| Parquet 小文件累积 | 按月/年分区写，避免按天；append 用 `pyarrow.ParquetWriter` 增量列追加，月底合并 |
| 内存膨胀 | deque 上限 1000 根，超出自动淘汰 |

### 6.3 并发模型

- 主进程跑 asyncio loop（WS / REST 都是 IO bound）
- 阻塞型工作（parquet flush / sqlite 批写）用 `run_in_executor` 推到线程池
- **不引入 multiprocessing**（Phase 1 单进程够用，跨进程通信复杂度不值）
