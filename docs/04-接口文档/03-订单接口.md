# 03 · 订单接口（OMS API）

> **稳定性**：**Beta v0.1**（Phase 3 实盘前可破坏式变更；Phase 4 上线前冻结为 Stable v1.0）
> **用途**：执行层 OMS 对**风控层 / 内部模块**提供的下单接口契约。**策略层不直接调用这套**——策略只产 Signal，Signal 经三层风控后才落到 OMS。
> **关联**：[详细设计/04-执行层](../03-详细设计/04-执行层.md) · [ADR-003 强制止损](../07-决策日志/ADR-003-强制交易所侧止损.md) · [数据模型 §3.2.2 orders](../05-数据模型.md#322-orders订单)

---

## 1. 用途

OMS 是**下单的唯一入口**。所有写入交易所的动作都必须通过 OMS：

- 三层风控通过后 → OMS 提交订单
- 强制止损单 → OMS 内部状态机自挂
- 多腿协调（S3）→ OMS 协调器
- 平仓 / 改单 / 撤单 → OMS

**调用方**：风控层 / Strategy Manager / TradeGroupCoordinator / 紧急平仓器（人工操作 Dashboard）。

---

## 2. 核心方法

```python
class OMS(Protocol):
    # ============ 下单 ============

    def place_order(self, intent: OrderIntent) -> OrderHandle: ...
    def place_orders_atomic(self, intents: list[OrderIntent], group_id: str) -> list[OrderHandle]: ...
    
    # ============ 改单 / 撤单 ============

    def cancel_order(self, client_order_id: str) -> CancelResult: ...
    def amend_order(self, client_order_id: str, **changes) -> OrderHandle: ...

    def cancel_all(self, symbol: str | None = None) -> list[CancelResult]: ...

    # ============ 紧急动作 ============

    def emergency_close(self, position_id: int, reason: str) -> OrderHandle: ...
    def emergency_close_all(self, reason: str) -> list[OrderHandle]: ...

    # ============ 查询 ============

    def get_order(self, client_order_id: str) -> Order | None: ...
    def get_open_orders(self, symbol: str | None = None) -> list[Order]: ...
    def get_position(self, symbol: str, strategy: str | None = None) -> Position | None: ...

    # ============ 模式控制 ============

    def get_mode(self) -> Literal['normal', 'degraded', 'frozen', 'paper']: ...
    def freeze(self, reason: str) -> None: ...
    def unfreeze(self, reason: str) -> None: ...      # 仅维护者可调
```

## 3. OrderIntent（下单参数）

OMS 接受的输入是"已通过 L1 风控的下单意图"：

```python
@dataclass
class OrderIntent:
    # ============ 来源溯源 ============
    signal_id: int                              # 关联 signals.id
    strategy: str                               # 'S1_btc_eth_trend'
    strategy_version: str                       # git short hash
    trade_group_id: str | None = None           # 多腿组（None=单腿）

    # ============ 订单本体 ============
    symbol: str                                 # 'BTC/USDT'
    side: Literal['buy', 'sell']
    order_type: Literal['limit', 'market', 'stop', 'stop_limit', 'take_profit']
    quantity: float                             # base asset 数量
    price: float | None = None                  # limit/stop_limit 必填，market 必空
    stop_price: float | None = None             # stop/stop_limit/take_profit 必填
    time_in_force: Literal['GTC','IOC','FOK','DAY'] = 'GTC'
    reduce_only: bool = False                   # 合约平仓用
    purpose: Literal['entry','stop_loss','take_profit','exit','hedge_leg'] = 'entry'

    # ============ 强制止损保护 ============
    stop_loss_price: float | None = None        # entry 单填这个，OMS 自动 5s 内挂止损

    # ============ 幂等 ============
    client_order_id: str                        # 必填（调用方生成 + 持久化），见 §3.1
```

### 3.1 字段约束

| 字段 | 约束 |
|---|---|
| `quantity` | 必须 ≥ symbol 的 `lot_size`，且对齐到 `lot_size` 整数倍 |
| `price` | 必须对齐到 `tick_size` 整数倍 |
| `quantity * price` | 必须 ≥ symbol 的 `min_notional` |
| `stop_loss_price` | 多头 < entry，空头 > entry，否则 raise `InvalidStopLoss` |
| `purpose='entry'` | **必须**带 `stop_loss_price`（除非 `reduce_only=True`） |
| `client_order_id` | **必填**，调用方生成 UUID v4 + 必须**先持久化到 `orders` 表后**再调 `place_order`（保证 retry 安全） |

**为什么强制调用方提供 `client_order_id`**：

OMS 的幂等性依赖于"重试时拿同一个 ID 重发"。如果 OMS 自动生成，调用方在网络超时 / 进程崩溃后无法知道刚才那个 ID 是什么——再调 `place_order` 等于发新单（重复下单 = 资金事件）。**唯一安全的方式是：调用方先生成 ID 并持久化到本地 SQLite `orders` 表（status='pending_submit'）**，崩溃恢复时按 ID 调 `query_order` 探测真实状态后再决定重发。

**测试 / Paper 便利模式**：`OMS(auto_generate_client_id=True)` 构造选项允许自动生成 ID，仅供 paper / unit test 使用；live 模式拒绝此选项。

## 5. 行为约定

### 5.1 幂等性（关键）

- 所有写操作以 `client_order_id` 为幂等 key
- 调用 `place_order(intent)` 时若 `client_order_id` 已存在：
  - 状态相同 → 返回原 OrderHandle，不重复发单
  - 状态不同（如已经 canceled）→ raise `IdempotencyConflict`
- 网络错误时调用方应**用同一 client_order_id 重试**（OMS 安全去重）

### 5.2 同步 vs 异步

| 方法 | 阻塞 | 何时返回 |
|---|---|---|
| `place_order` | 半阻塞 | 等到 ACK 或 reject（≤ 5s）；超时 raise `SubmitTimeout` |
| `place_orders_atomic` | 半阻塞 | 全部 ACK 或全部回滚 |
| `cancel_order` | 阻塞 | 等到交易所确认（≤ 3s） |
| `amend_order` | 阻塞 | 同上 |
| `emergency_close` | 阻塞 | 等到平仓单提交（不等成交） |

### 5.3 强制止损契约

`OrderIntent` 携带 `stop_loss_price` 时，OMS 必须保证：

1. 该 entry 单 fill 后，**5 秒内**挂出对应的交易所侧 stop_loss 单
2. 9 类失败分支按 [详细设计/04-执行层 §4.4](../03-详细设计/04-执行层.md#44-9-类失败分支处置矩阵) 处理
3. 失败到 AUTO_CLOSE 时，OMS 立即发市价平仓单 + critical 告警

调用方**不需要**关心止损细节——只要传 `stop_loss_price`，OMS 全权保护。

### 5.4 多腿原子化

`place_orders_atomic(intents, group_id)`：

- 所有 intent 必须共享同一个 `trade_group_id`
- 全部成功 → 返回 N 个 handle，trade_group status `forming → hedging`
- 任一失败 → **撤所有已成功的**，trade_group status `broken`
- 30s 超时未全成 → 同上

详细见 [详细设计/04-执行层 §5](../03-详细设计/04-执行层.md#5-tradegroupcoordinator双腿原子化)。

### 5.5 模式开关

| 当前模式 | 接受新单？ | 处理已挂单？ |
|---|---|---|
| `normal` | ✅ | ✅ |
| `degraded` | ❌ 拒绝（`OMSDegraded` 异常） | ✅ 继续 |
| `frozen` | ❌ | ❌ 拒绝改单 / 撤单（除 emergency_close 外） |
| `paper` | ✅（mock 撮合） | ✅ |

`unfreeze` 必须维护者本人调用（通过 Dashboard 确认按钮，或本地 CLI）；不允许策略 / Agent 自动解冻。

## 6. 错误码

| 异常类 | 场景 | 调用方处置 |
|---|---|---|
| `OrderRejected` | 交易所拒单（reject reason 在异常 message） | 落 risk_events，不重试 |
| `RateLimited` | API 限流（429 / -1003） | 退避重试（OMS 内已实现） |
| `IdempotencyConflict` | 同 client_order_id 但状态冲突 | 查 order 状态后决定 |
| `SubmitTimeout` | 5s 内未收到 ACK | 用同 client_order_id 查询；不要直接重发 |
| `InvalidOrderIntent` | 字段不合法（精度 / 必填项 / stop 价位） | 编程错误，修代码 |
| `InvalidStopLoss` | stop 价位与方向不匹配 | 同上 |
| `OMSDegraded` | OMS 在 degraded / frozen 模式 | 风控层应早判断，不该到这一步 |
| `PositionModeMismatch` | hedge mode vs one-way mode 不一致 | OMS 拒绝启动；维护者修配置 |
| `ReconcileAnomaly` | 本地 vs 交易所对账失败 | OMS 自动 freeze；critical 告警 |

### 6.1 错误码与交易所原始码映射

OMS 维护一个 `ExchangeErrorMap`，把 Binance / OKX 原始错误码归一化到上面的异常类。详见 `core/execution/error_map.py`（待实现）。

## 7. 调用示例

### 7.1 风控层下普通单

```python
intent = OrderIntent(
    signal_id=signal.id,
    strategy=signal.strategy,
    strategy_version=signal.strategy_version,
    symbol='BTC/USDT',
    side='buy',
    order_type='market',
    quantity=l2_sizer_output,            # 经过 L2 计算
    stop_loss_price=signal.stop_price,   # **必填**
    purpose='entry',
)
try:
    handle = oms.place_order(intent)
except OrderRejected as e:
    log_risk_event('order_rejected', signal_id=signal.id, reason=str(e))
except SubmitTimeout:
    existing = oms.get_order(intent.client_order_id)
    ...
```

### 7.2 S3 双腿原子下单

```python
group_id = uuid4().hex
spot_intent = OrderIntent(
    ..., symbol='BTC/USDT', side='buy', order_type='market',
    purpose='hedge_leg', trade_group_id=group_id,
)
perp_intent = OrderIntent(
    ..., symbol='BTC/USDT:USDT', side='sell', order_type='market',
    purpose='hedge_leg', trade_group_id=group_id,
)
handles = oms.place_orders_atomic([spot_intent, perp_intent], group_id)
```

### 7.3 紧急平仓

```python
oms.emergency_close(position_id=42, reason='manual_dashboard_button')
```

## 8. 性能与可靠性

- 单次 `place_order` 端到端 P99 ≤ 500ms（不含交易所撮合）
- OMS 持久化所有写操作到 SQLite（事务 + WAL）
- 进程崩溃后重启：扫描 SQLite 中 `status='new'/'partial'` 的订单 + 调交易所 API 对账，重建状态
- 不允许"在内存里飞"的订单——任何已发出的下单意图必须先落 orders 表
