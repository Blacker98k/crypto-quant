## 写作约定

每份接口文档遵循：

```markdown
# 接口名

## 用途
一句话讲清这个接口干啥。

## 签名
（参数、返回值、抛出异常）

## 行为约定
- 幂等性
- 阻塞 / 非阻塞
- 超时
- 重试策略

## 调用示例
（伪代码或核心调用模式）

## 错误码 / 异常
| code | 含义 | 处理 |

## 版本
v1.0 (2026-XX-XX)
变更历史：v0.X → v1.0
```

## 接口稳定性等级

| 等级 | 含义 | 例子 |
|---|---|---|
| **Stable** (v1.0+) | 公开稳定接口，向后兼容；破坏式变更需 ADR + 兼容期 | `Strategy.on_bar()`（Phase 4 前冻结） |
| **Beta** (v0.x) | 尚未冻结，**允许破坏式变更**，但每次破坏必须**同步更新所有调用方 + 文档 + 此 README 状态** | 当前所有 v0.1 接口 |
| **Internal** | 模块内部使用，可随意改，不构成对外契约 | `OMS._reconcile()` |

**Beta → Stable 升级条件**（Phase 4 上线前必走）：

1. 接口已经过 Phase 3 的 OMS 闭环 + 策略实现 + paper 30 天的实战验证
2. 没有发现需要破坏式变更的设计缺陷
3. 升 Stable v1.0 + 写一份 ADR 记录冻结快照

**Stable 之后的破坏性变更**：

1. 先标 `@deprecated` 一个版本
2. 写迁移指南
3. 下版本才移除
