## 2. ExchangeAdapter（交易所适配）

### 2.1 抽象基类

`core/data/exchange/base.py`：

```python
class ExchangeAdapter(Protocol):
    name: str                          # 'binance_spot' / 'binance_usdm' / 'okx_spot' ...
    market_type: Literal['spot','perp','futures']

    # ============ 元数据 ============
    async def fetch_exchange_info(self) -> list[SymbolInfo]: ...
    async def fetch_24h_tickers(self) -> list[Ticker24h]: ...

    # ============ 历史 ============
    async def fetch_klines(
        self, symbol: str, timeframe: str,
        start_ms: int, end_ms: int, limit: int = 1000,
    ) -> list[Bar]: ...

    async def fetch_funding_rates(
        self, symbol: str, start_ms: int, end_ms: int,
    ) -> list[FundingRate]: ...     # 仅 perp adapter 实现

    async def fetch_open_interest(
        self, symbol: str, start_ms: int, end_ms: int,
    ) -> list[OpenInterest]: ...    # 仅 perp adapter 实现

    # ============ 实时（WS） ============
    async def subscribe_klines(
        self, symbols: list[str], timeframes: list[str],
        on_bar: Callable[[Bar], Awaitable[None]],
    ) -> SubscriptionHandle: ...

    async def subscribe_funding(
        self, symbols: list[str],
        on_funding: Callable[[FundingRate], Awaitable[None]],
    ) -> SubscriptionHandle: ...

    # ============ 下单（Phase 3a 才实现）============
    async def place_order(self, payload: OrderPayload) -> OrderAck: ...
    async def cancel_order(self, symbol: str, exchange_order_id: str) -> CancelAck: ...
    async def query_order(self, symbol: str, exchange_order_id: str) -> OrderState: ...
    async def fetch_open_orders(self, symbol: str | None = None) -> list[OrderState]: ...
    async def fetch_positions(self) -> list[ExchangePosition]: ...
    async def fetch_balance(self) -> Balance: ...
```

### 2.2 Binance 实现

| 子类 | base_url | WS endpoint | 用途 |
|---|---|---|---|
| `BinanceSpotAdapter` | `https://api.binance.com` | `wss://stream.binance.com:9443/ws` | 现货 |
| `BinanceUsdmAdapter` | `https://fapi.binance.com` | `wss://fstream.binance.com/ws` | U 本位永续 |

**国内访问**：必须走代理。代理来自配置 `config/runtime.yml`：

```yaml
exchange:
  proxy: "http://127.0.0.1:57777"   # 可空；空则走系统代理
  request_timeout_ms: 5000
  max_retries: 3
```

### 2.3 速率限制（Rate Limit）

| 类型 | Binance Spot 限额 | Binance USDM 限额 | 处置 |
|---|---|---|---|
| REQUEST_WEIGHT | 6000/min | 2400/min | 全局令牌桶限速 |
| ORDERS | 100/10s 单 IP | 300/10s 单 IP | 同上 |
| RAW_REQUESTS | 61000/5min | 同 spot | 同上 |

**实现要点**：

- 每个 adapter 内置令牌桶，按 endpoint 的 `weight` 扣令牌
- 收到 `429` / Binance `-1003` 立即暂停 + 按响应头 `Retry-After` 退避
- 被 ban（HTTP 418）→ 立即停所有 REST + critical 告警 + 维护者必须人工解封
- WS 不计入 REQUEST_WEIGHT，但单连接 max 200 stream / max 5 incoming msg/s 反向

### 2.4 WS 行为约定

| 行为 | 规则 |
|---|---|
| 心跳 | 客户端每 30s ping；3 分钟无 pong 视为断 |
| 单连接 stream 数 | ≤ 200；超出拆多连接 |
| 重订阅 | 同 stream 不重复订阅；重复 → 上层 raise `DuplicateSubscription` |
| 断线 | 自动重连，指数退避（1s / 2s / 5s / 10s / 30s 上限）+ 重订阅原 stream |
| 缺数据 | 重连后 **必须** 调 REST 补"断线期间错过的 closed bar" |
| 推送的 partial bar | 默认丢弃（仅推 closed=True）；上层显式订阅 partial 才转发 |

### 2.5 凭据

| 文件 | 字段 | 权限 |
|---|---|---|
| `config/secrets.yml` | `binance_spot.api_key` `binance_spot.api_secret` `binance_usdm.api_key` `binance_usdm.api_secret` | 600 |
| `config/secrets.example.yml` | 占位（提交到 git） | 644 |

**约束**：

- secrets.yml 永远不入 git（[ADR-009 §2.3](../07-决策日志/ADR-009-远程Agent与可信代码边界.md) + `.gitignore`）
- adapter 启动时打印前 4 / 后 4 字符脱敏：`abcd****wxyz`
- Phase 1 不读 api_key（只用公开 REST/WS）

## 3. Telegram Bot

> **Phase 1 不接入**。本节为 Phase 3a 准备的契约。

### 3.1 用途

把**关键事件**主动推到维护者手机：开仓 / 平仓 / 熔断 / 止损失败 / 网络异常 / 对账异常。

### 3.2 配置

```yaml
# config/runtime.yml
telegram:
  enabled: false              # Phase 3a 才打开
  bot_token: "${TELEGRAM_BOT_TOKEN}"   # 从 env 读，secrets.yml 不存
  chat_id: 123456789
  rate_limit_per_min: 30      # 防消息刷屏
```

### 3.3 推送规则

| 事件 | 优先级 | 频次限制 |
|---|---|---|
| 服务启动 / 停止 | info | - |
| 单笔开仓 / 平仓 | info | 同 symbol 同 side ≥ 30s 才再推 |
| 单日盈亏 | info | 每日 09:00 / 21:00 推一次 |
| 熔断触发（L3） | critical | 不限频 |
| 强制止损失败（AUTO_CLOSE） | critical | 不限频 |
| 对账异常 | critical | 同 symbol 5min 一次 |
| WS 全断 ≥ 60s | critical | 一次 |

### 3.4 错误处置

| 场景 | 处置 |
|---|---|
| Telegram API 不通 | 退避重试 3 次；仍失败 → 落 risk_events，**不阻塞主流程** |
| 消息被限流（429） | 进入 cooldown 60s |
| 维护者 chat_id 错误 | 启动时探测一次，错误 → 拒绝启动 + 打印明显错误 |

### 3.5 接口

```python
class Notifier(Protocol):
    async def send(self, level: Literal['info','warn','critical'], msg: str, **fields) -> None: ...
    async def send_event(self, event: RiskEvent) -> None: ...
```

调用方：执行层 / 风控层 / Dashboard。

## 4. Dashboard REST

> Dashboard 内部实现见 [详细设计/06-监控面板](../03-详细设计/06-监控面板.md)（待编写）。本节定义对外 HTTP API。

### 4.1 部署

- **本机绑定** `127.0.0.1:8088`（不暴露公网）
- 不带认证（依赖网络层隔离）；如需远程访问走 SSH 隧道
- `no_proxy` 必须含 `127.0.0.1,localhost`（[踩坑记录 - 全局代理](../09-踩坑记录.md)）

### 4.2 路由（Phase 1 最小集）

| 方法 | 路径 | 用途 | 返回 |
|---|---|---|---|
| GET | `/health` | 健康自检 | `{status: 'ok' \| 'degraded' \| 'down', checks: {...}}` |
| GET | `/api/symbols?layer=core` | 列币池 | `[{symbol, type, universe}]` |
| GET | `/api/candles?symbol=BTCUSDT&tf=1h&limit=200` | 取 K 线 | `[{ts, o, h, l, c, v}]` |
| GET | `/api/ticker/BTCUSDT` | 最新价 | `{symbol, price, ts}` |
| GET | `/api/positions` | 当前持仓（paper） | `[{symbol, side, qty, avg_entry, unrealized_pnl}]` |
| GET | `/api/run_log?since=ts&limit=100` | 接口健康日志 | `[{endpoint, status, http_code, latency_ms, captured_at}]` |

### 4.3 Phase 3a 追加的路由

| 方法 | 路径 | 用途 |
|---|---|---|
| GET | `/api/orders?status=open` | 当前活跃订单 |
| GET | `/api/risk_events?severity=critical&since=ts` | 风险事件 |
| GET | `/api/strategies` | 各策略状态 / 收益 |
| GET | `/api/trade_groups?status=hedged` | 多腿持仓状态 |
| POST | `/api/freeze` | 维护者本人触发熔断（带二次确认 token） |
| POST | `/api/unfreeze` | 解冻（带二次确认 token） |
| POST | `/api/emergency_close/{position_id}` | 紧急平仓（带二次确认） |

**写操作约束**：

- 任何 POST 必须带 `X-Confirm-Token: <session-token>`，token 由 Dashboard 启动时打印到本机日志
- 写操作落 audit 表（不实现的话至少落 `risk_events` source='dashboard'）

### 4.4 响应规范

```json
// 成功
{
  "ok": true,
  "data": <payload>,
  "meta": {"ts": 1735689600000, "version": "0.1.0"}
}

// 失败
{
  "ok": false,
  "error": {"code": "DataNotAvailable", "message": "..."},
  "meta": {...}
}
```

HTTP 状态码：成功 200；客户端错误 4xx；服务器错误 5xx。

### 4.5 性能要求

- `/api/candles?limit=200` P99 < 200ms
- `/api/health` < 50ms
- 同时支持 ≥ 10 个并发 WebSocket Dashboard 连接（Phase 2 引入推送）

## 5. 错误码归一化

各家交易所原始错误码 → 项目自定义异常的映射表，由 `core/execution/error_map.py` 维护。

### 5.1 异常类层级

```
CryptoQuantError
├── DataFeedError                # 数据层
│   ├── DataNotAvailable
│   ├── DataTimeout
│   └── ParquetCorrupt
├── ExchangeAPIError             # 外部接口
│   ├── RateLimited              # 429 / -1003
│   ├── IPBanned                 # 418
│   ├── AuthError                # 签名失败 / api_key 无效
│   ├── NetworkError             # 超时 / DNS / 连接拒绝
│   └── ExchangeServerError      # 5xx
├── OrderRejectedError           # 下单被拒
│   ├── InvalidOrderIntent       # 字段不合法（精度等）
│   ├── InsufficientBalance
│   ├── PositionModeMismatch
│   └── ReduceOnlyRejected
└── ReconcileAnomaly
```

### 5.2 Binance 错误码映射（高频）

| Binance code | Python 异常 | 调用方处置 |
|---|---|---|
| HTTP 429, body `-1003` | `RateLimited` | 退避（OMS 内已实现），上层不处理 |
| HTTP 418 | `IPBanned` | critical 告警 + 全局停所有 REST + 等维护者人工 |
| `-2010 NEW_ORDER_REJECTED` | `OrderRejectedError` | 落 risk_events，不重试 |
| `-2011 CANCEL_REJECTED` | `OrderRejectedError`(reason='already_done') | 查 order 状态，多半已 fill |
| `-2013 NO_SUCH_ORDER` | `OrderRejectedError`(reason='not_found') | 查本地是否漏发，可能需要重发 |
| `-1013 INVALID_QUANTITY` / `-1111 BAD_PRECISION` | `InvalidOrderIntent` | 编程错误，修代码（lot_size / tick_size 没对齐） |
| `-1102 MANDATORY_PARAM_EMPTY` | `InvalidOrderIntent` | 同上 |
| `-2018 BALANCE_NOT_SUFFICIENT` | `InsufficientBalance` | L2 风控 sizer 应早判断；落 risk_events |
| `-4061 POSITION_SIDE_NOT_MATCH` | `PositionModeMismatch` | OMS 拒绝启动；维护者改配置 |
| `-2022 REDUCE_ONLY_REJECT` | `ReduceOnlyRejected` | 见 [执行层 §4.4 失败 #8](../03-详细设计/04-执行层.md#44-9-类失败分支处置矩阵) |
| `-1021 TIMESTAMP` | `AuthError`(reason='clock_skew') | 同步系统时间；持续不修 → 拒绝启动 |
| `-2014 / -2015 SIGNATURE` | `AuthError` | secret 配置错；告警 + 拒绝启动 |
| HTTP 5xx | `ExchangeServerError` | 退避重试 3 次 |

完整表见 `core/execution/error_map.py` 的字典常量（待 Phase 3a 实现时填）。

### 5.3 调用方契约

- 调用 `ExchangeAdapter` 的代码**只 catch 上面的异常类**，不要 catch 原始 HTTP/库异常
- 任何未归类的异常视为 bug，立即 raise + critical 告警

## 6. 健康自检

每个外部接口都注册到 `HealthChecker`：

```python
class HealthCheck(Protocol):
    name: str                       # 'binance_spot_rest' / 'binance_usdm_ws' / 'telegram' / ...
    async def check(self) -> HealthResult: ...

@dataclass
class HealthResult:
    name: str
    status: Literal['ok', 'degraded', 'down']
    latency_ms: int | None
    last_ok_ts: int | None
    detail: str | None
```

**频率**：每 30s 跑一次，结果落 `run_log`；连续 3 次 down 触发告警。

## 7. 安全约束

| 类别 | 约束 |
|---|---|
| API key | 仅 `secrets.yml`，权限 600，**永不入 git** |
| 日志 | 不打印完整 key/token；脱敏 `abcd****wxyz` |
| Dashboard | 仅本机绑定；不暴露公网；写操作 token 校验 |
| Telegram | bot_token 走 env，**不进 secrets.yml**（避免备份/同步意外泄露） |
| 错误信息 | 异常 message 不带任何凭据片段 |
