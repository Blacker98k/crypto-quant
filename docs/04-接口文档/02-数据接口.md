# 02 · 数据接口

> **稳定性**：**Beta v0.1**（Phase 3 实盘前可破坏式变更；Phase 4 上线前冻结为 Stable v1.0）
> **用途**：策略与回测引擎读取行情数据的统一接口。**研究模式从 Parquet 读、生产模式从 WS+内存读，对调用方完全透明**。
> **关联**：[架构 §4 数据流](../02-架构设计文档.md#4-数据流) · [详细设计/01-数据层](../03-详细设计/01-数据层.md)（待编写）· [数据模型 §3.1 共享实体](../05-数据模型.md#31-共享实体研究--生产)

---

## 1. 用途

`DataFeed` 是策略 / 回测 / Dashboard 唯一访问行情的入口。底层封装 Parquet / DuckDB / 内存缓存 / WS 订阅，对外提供两种模式：

- **拉取模式**：`get_*(symbol, tf, start, end)` 同步返回历史数据
- **推送模式**：`subscribe_*(callback)` 异步推送实时数据

策略代码用同一份 `DataFeed`，框架根据运行模式（research / live / paper）注入不同的实现。

---

## 2. DataFeed 接口

```python
class DataFeed(Protocol):
    # ============ 拉取模式（历史 / 当前快照） ============

    def get_candles(
        self,
        symbol: str,
        timeframe: str,
        start_ms: int | None = None,
        end_ms: int | None = None,
        n: int | None = None,
    ) -> list[Bar]: ...

    def get_funding_rates(
        self,
        symbol: str,
        start_ms: int | None = None,
        end_ms: int | None = None,
    ) -> list[FundingRate]: ...

    def get_orderbook_snapshot(
        self,
        symbol: str,
        depth: int = 5,
    ) -> OrderBook: ...                   # 实时模式才可用

    def get_last_price(self, symbol: str) -> float: ...

    # ============ 推送模式（实时） ============

    def subscribe_candles(
        self,
        symbol: str,
        timeframe: str,
        callback: Callable[[Bar], None],
    ) -> SubscriptionHandle: ...

    def subscribe_ticker(
        self,
        symbol: str,
        callback: Callable[[Tick], None],
    ) -> SubscriptionHandle: ...

    def subscribe_funding(
        self,
        symbol: str,
        callback: Callable[[FundingRate], None],
    ) -> SubscriptionHandle: ...

    def unsubscribe(self, handle: SubscriptionHandle) -> None: ...

    # ============ 元数据 ============

    def get_symbol_info(self, symbol: str) -> SymbolInfo: ...
    # tick_size / lot_size / min_notional / contract_type ...

    def list_universe(self, layer: Literal['core','observation','scan']) -> list[str]: ...
```

## 3. 时间语义（关键）

### 3.1 区间约定

`get_candles(start_ms=A, end_ms=B)` 返回的 K 线满足：

```
A ≤ bar.ts < B
```

**左闭右开**。`bar.ts` 是开盘时间。

### 3.2 `n` 与 `start/end` 互斥

- `n` 不为 None → 返回距 `now_ms()`（含回测注入）最近的 n 根**已收盘** K 线
- `start_ms` / `end_ms` 至少一个不为 None → 按区间返回

两组参数不能混用，混用会 raise `InvalidQueryError`。

### 3.3 已收盘 vs 进行中

- `get_*()` 返回的 Bar 默认 `closed=True`
- `subscribe_candles()` 默认只推 closed=True
- 想要进行中 K 线必须 `subscribe_candles(.., partial=True)`

### 3.4 时区

- 所有时间戳一律 UTC 毫秒（INTEGER）
- 接口不接受 datetime / 字符串
- 显示时上层自己做时区转换

## 4. 缺数据处理

| 场景 | 行为 |
|---|---|
| 区间内有 K 线缺失（数据洞） | 返回已有的，附带 `meta.missing_ranges`；调用方自行判断是否能继续 |
| 完全没有该 symbol/timeframe 数据 | raise `DataNotAvailable` |
| `n=200` 但只有 50 根 | 返回 50 根，**不补 0** |
| 实时订阅时 WS 中断 | 自动重连 + 用 REST 补漏；恢复后回放期间错过的 closed=True bar |

**不允许默默补 0 / 前向填充**——策略要么有数据，要么明确知道缺数据。

## 5. 订阅模型

### 5.1 SubscriptionHandle

```python
@dataclass
class SubscriptionHandle:
    id: str
    symbol: str
    stream: str            # 'candles_4h' / 'ticker' / 'funding'
    callback: Callable
    state: Literal['active', 'paused', 'closed']
```

### 5.2 回调时序

- 单一 symbol+stream 的 callback **严格按时间顺序**调用
- 跨 symbol 的回调可能并发；策略代码应当幂等
- callback 内**禁止阻塞**：≤ 50ms 必须返回，长任务自己丢 queue

### 5.3 Backpressure

如果 callback 慢于推送速度：

- DataFeed 维护一个**有界队列**（默认 1000 条）
- 满了直接**丢弃最旧**（不是阻塞 producer）
- 丢弃事件落 `run_log` + warn 告警
- 策略可以配置 `slow_callback_policy='block' | 'drop_oldest' | 'drop_newest'`

## 6. 模式差异（透明但记录）

| 调用方法 | research 模式 | live 模式 | paper 模式 |
|---|---|---|---|
| `get_candles(start, end)` | 从 Parquet 直读 | 从 SQLite 读已落库 + 内存读未落库 | 同 live |
| `get_last_price()` | bar.c 模拟值 | WS 内存 | WS 内存（与 live 共享） |
| `subscribe_candles()` | 按 ts 顺序回放历史 | 真实 WS 推送 | 真实 WS 推送（不下单） |
| `get_orderbook_snapshot()` | **不可用**（raise NotInLiveMode） | WS 推送 | 同 live |

**含义**：`get_orderbook_snapshot` 这类只在实时模式有的方法，策略调用前必须 try/except 或 capability 检测。建议尽量不依赖订单簿做信号（回测无法重现）。

---

## 7. 行为约定

### 7.1 幂等性

`get_candles` 在相同参数下必须返回相同结果（在底层数据未更新前提下）。

### 7.2 缓存策略

- DataFeed 内部可以缓存最近 N 根 K 线
- 但**不暴露缓存 invalidate API**——上层不应感知缓存
- DuckDB / Parquet 自身的查询缓存由其管理

### 7.3 超时

- `get_*()` 默认 5s 超时；超时 raise `DataTimeout`
- `subscribe_*` 不阻塞，立即返回 handle

### 7.4 错误码

| 异常 | 何时 | 处置 |
|---|---|---|
| `DataNotAvailable` | 该 symbol/tf 完全没数据 | 策略 on_bar 应跳过 |
| `InvalidQueryError` | n 与 start/end 混用 | 编程错误，需修代码 |
| `NotInLiveMode` | research 模式调实时方法 | 策略应做 capability 判断 |
| `DataTimeout` | 拉数据超时 | 重试 1 次或跳过 |

## 8. 使用示例

```python
def on_bar(self, bar: Bar, ctx: StrategyContext) -> list[Signal]:
    history = ctx.data.get_candles(
        symbol=bar.symbol,
        timeframe='1h',
        n=200,
    )
    if len(history) < 100:
        return []

    last_funding = ctx.data.get_funding_rates(
        symbol=bar.symbol,
        start_ms=ctx.now_ms() - 7 * 86400_000,
    )
    ...
```
