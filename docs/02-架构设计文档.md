## 1. 设计原则

四条不可妥协的原则，所有后续决策围绕它们：

### 1.1 Research-Production Parity（研究-生产一致性）

**研究区跑出的策略代码不改一行就能上生产**。

为什么：业内最大的坑是"回测赚钱、实盘亏钱"。原因 80% 是研究和生产用了**不同的逻辑**——回测的撮合假设、数据对齐、特征生成与实盘不一致。

怎么做：
- 回测的 `paper 撮合器` 和实盘的 `paper 模式`是**同一个类**
- 历史 K 线（研究）和实时 K 线（生产）schema 完全一致
- `Strategy` 是统一接口，不区分回测/实盘

### 1.2 三层风控冗余

任何一层失效，其他两层兜底。

| 层 | 时机 | 失败后果 |
|---|---|---|
| L3 投资组合层 | 信号产生后 | 拒绝信号 |
| L2 仓位层 | 信号通过后 | 缩小或拒绝订单 |
| L1 订单层 | 订单提交前 | 取消订单或追单 |

详见 [§6 风控体系](#6-风控体系)。

### 1.3 强制交易所侧止损

**任何持仓必须在交易所侧挂止损单**。

为什么：本地服务可能宕机、断网、被挂起。只有交易所侧的止损单永远生效。

代价：可能被插针扫掉一些利润，但**极端行情时是救命的**。这是不可妥协的。

### 1.4 Paper 优先

**任何策略实盘前 paper 跑 ≥ 30 天**。

为什么：参数过拟合、regime change、流动性陷阱在 30 天内大概率会暴露。实盘前不验证 = 用真钱做随机实验。

## 2. 总体架构

### 2.1 双区域结构

系统**水平分两个区域**，通过共享数据层和共享接口连接：

```
┌─────────────────────────────────────────────────┐
│            Research Platform 研究区              │
│  - 因子库、回测引擎、策略实验室                    │
│  - 输出：评分卡、HTML 报告、推荐参数              │
└─────────────────────────────────────────────────┘
              ↓ 通过 OOS 验证的策略
┌─────────────────────────────────────────────────┐
│            Trading System 生产区                 │
│  - 数据采集、信号生成、风控、订单执行              │
│  - 输出：持仓、PnL、风险事件                      │
└─────────────────────────────────────────────────┘
              ↑↓
        共享：Strategy 接口 / DB Schema / Config
```

**关键**：两区域用同一个 `Strategy` 接口和同一份数据，仅在"是否真发订单"上分叉。

### 2.2 五大模块（生产区）

```
数据层 → 策略层 → 风险层 → 执行层
                                ↓
                         监控层（横向贯穿）
```

| 模块 | 职责 | 输入 | 输出 |
|---|---|---|---|
| 数据层 | 拉行情、持久化、提供统一接口 | 交易所 API | 标准化的 K 线/订单簿/资金费 |
| 策略层 | 评估机会、产生信号 | 数据 + 因子 | 候选信号（Signals） |
| 风险层 | 三层过滤 | 候选信号 | 通过/拒绝、仓位大小 |
| 执行层 | 订单管理、滑点控制 | 通过的信号 | 订单/成交/持仓 |
| 监控层 | Dashboard / 告警 / 日志 | 全链路事件 | 可观测性 |

### 2.3 三个研究子模块

| 模块 | 职责 |
|---|---|
| 因子库 | 沉淀可复用的特征计算（动量、波动、量价、资金面） |
| 回测引擎 | 历史数据上重放策略，输出收益曲线 |
| 策略实验室 | 参数扫描、Walk-forward、A/B 比较、OOS 验证 |

## 3. 核心抽象

### 3.1 Strategy 接口

所有策略实现统一接口，研究和生产共用：

> 接口签名细节见 [`docs/04-接口文档/01-策略接口.md`](./04-接口文档/01-策略接口.md)

核心方法：
- `required_data()` 声明依赖（哪些币、哪些时间段、哪些指标）
- `on_bar()` K 线收盘时被调用，返回 0..N 个信号
- `on_tick()` 可选，tick 级响应（做市/触发器）
- `on_fill()` 可选，成交回调

### 3.2 Signal 数据模型

策略产出的标准信号格式，包含：
- 标的、方向（做多/做空/平仓）
- 入场价、止损价、目标价
- 置信度、建议仓位
- 决策依据（rationale）

不是订单，是**意图**。订单由风控+执行层根据信号生成。

### 3.3 DataFeed 接口

策略不直接访问 DB 或交易所，统一通过 `DataFeed`：
- 历史模式：从 parquet/sqlite 读
- 实时模式：从 WS / 内存缓存读
- 接口完全一致，策略代码无感

### 3.4 OMS（订单管理系统）

执行层的核心抽象。状态机：

```
NEW → PENDING_NEW → PARTIALLY_FILLED → FILLED
              ↘ REJECTED / CANCELED / EXPIRED
```

支持：限价 / 市价 / 止损 / 止盈 / IOC / FOK，以及部分成交追单。

### 3.5 Strategy Manager（组合管理器）

不是简单把 N 个策略 PnL 相加，而是：
1. **信号去重**：同币同方向多策略发信号 → 合并
2. **加权**：根据各策略历史 sharpe + 当前置信度
3. **相关性控制**：策略间相关系数 > 0.7 自动减仓
4. **敞口分配**：核心池/观察池/普查池按 70/20/10 分配

## 4. 数据流

### 4.1 实时数据流（生产）

```
Binance WS  ──→  数据层 (内存缓存 + 落库)
                    │
                    ├──→ 策略层（订阅推送）
                    └──→ Dashboard（推送显示）
```

**特点**：
- 推模式（WS 主动推，不轮询）
- 内存中保留近 N 根 K 线供策略快速访问
- 异步落库（不阻塞策略）

### 4.2 历史数据流（研究）

```
交易所 REST  ──→  Backfill Job  ──→  Parquet/SQLite
                                          │
                                          ↓
                                   回测引擎（向量化加载）
```

**特点**：
- 一次性回填 + 增量更新
- 列式存储（parquet）压缩比高、查询快
- DuckDB 直接查 parquet 不需要导入

### 4.3 数据一致性原则

- 时间戳统一**毫秒级 UTC**
- 价格精度遵循交易所规则（每个币不同）
- K 线对齐**自然时间**（非用户启动时间）
- 历史和实时 schema 完全一致

## 5. 信号-决策-执行链路

完整的从"市场行情"到"成交回报"的链路。**风控三层全部在 OMS 提交订单之前完成**：

```
[市场行情]
    │
    ↓
[数据层] 解码 / 聚合 / 落库 / 推送
    │
    ↓
[策略层] 评估机会
    │
    ↓
[Signal] 候选信号（落 signals 表，status=pending）
    │
    ↓
[Strategy Manager] 多策略信号合成（去重 / 加权 / 相关性控制）
    │
    ↓
[风险层 L3] 投资组合层检查（总杠杆 / 单币种敞口 / MDD / 熔断）
    │ ├── 拒绝 → 落 risk_events，signals.status=rejected
    │ └── 通过 ↓
[风险层 L2] 仓位层（ATR sizer / 单笔风险 / 强平距离 / Kelly）
    │ ├── 输出仓位为 0 → 拒绝
    │ └── 输出建议仓位 ↓
[风险层 L1] 订单层（滑点估计 / 重复信号去重 / 订单合法性 / tick_size & lot_size）
    │ ├── 拒绝 → 落 risk_events
    │ └── 通过 ↓
[执行层 OMS] ←─────────────  提交订单前必须三层全过  ─────────────
    │
    │ 内部子状态机：
    │  1. 生成 client_order_id（幂等 key）
    │  2. 提交订单 → 落 orders 表（status=new）
    │  3. 等待 ack
    │  4. 开仓成功后 ≤ 5s 内挂交易所侧止损单（原子保护）
    │     │ 失败 → 触发自动平仓（详见 03-详细设计/04-执行层）
    │
    ↓
[交易所] 撮合 → 推送成交
    │
    ↓
[执行层] 处理成交 → 更新持仓 → 落 fills 表 → 同步止损单数量
    │
    ↓
[监控层] Dashboard / Telegram / 日志
```

**关键约束（钉死实现）**：
- 三层风控**全部在 OMS 之前**，不允许"先发单再检查"
- 强制止损单是 **OMS 内部状态机**，不属于 L1 风控规则
- 每一步的拒绝都落 `risk_events`，方便事后复盘
- L1 → OMS 传递的是"已校准的下单意图"，OMS 不再做风控判断

详细的 OMS / 止损状态机 / 失败分支见 [`03-详细设计/04-执行层.md`](./03-详细设计/04-执行层.md)。

## 6. 风控体系

### 6.1 三层风控

#### L3 投资组合层（信号过滤）

| 规则 | 阈值 | 触发动作 |
|---|---|---|
| 总杠杆 | ≤ 3x | 新信号被拒绝 |
| 单币种敞口 | ≤ 25% 总资金 | 同币种新信号被拒绝 |
| 单日浮亏 | ≥ 5% | 熔断暂停 24h |
| 最大回撤 | ≥ 20% | **强制清仓 + 24h 停机** |
| 连续亏损单数 | ≥ 5 | 减半仓位 24h |

#### L2 仓位层（仓位计算）

- ATR-based 仓位：`position = risk_amount / (atr × atr_multiplier)`
- 单笔风险 ≤ 1% 总资金
- 强平距离 ≥ 30%（合约）
- Kelly fraction 0.25（不全仓 Kelly）

#### L1 订单层（订单提交交易所**之前**）

L1 只做**单笔订单合法性 + 即时市场状况**检查，不修改信号意图：

- 滑点估计：市价单预估滑点 > 0.2% → 拒绝或转为限价
- 重复检查：同 strategy + symbol + side 在 5s 内重复信号 → 去重
- 订单合法性：价格对齐 `tick_size`、数量对齐 `lot_size`、订单价值 ≥ `min_notional`
- 异常市场状况：买卖盘价差过宽 / 深度不足 → 拒绝

**强制止损单不是 L1 规则**，而是 OMS 内部状态机（开仓后 5s 内挂止损是原子保护行为）。详见详设/04-执行层。

部分成交追单（"5min 内未完全成交则改价或取消"）也属于 OMS 内部行为，不是 L1 风控规则。

### 6.2 熔断机制

熔断不是"暂停下单"，是**主动平仓 + 拒绝新信号**：

| 触发 | 动作 |
|---|---|
| 单日亏 5% | 平掉浮亏 > 1% 的仓位，拒绝 24h 内所有新信号 |
| 总回撤 20% | **全部清仓**，拒绝 24h 内所有新信号，发警报 |
| 连续 5 笔亏损 | 单笔风险减半（1% → 0.5%）持续 24h |
| API 异常率 > 10% / 5min | 暂停下单（不平仓），告警 |

### 6.3 三层冗余的意义

任何一层都可能有 bug。冗余设计：

| 场景 | L3 拦截 | L2 拦截 | L1 拦截 |
|---|---|---|---|
| 策略发了爆杠杆信号 | ✅ 拒绝 | - | - |
| L3 算错了让信号通过 | ❌ | ✅ 仓位算到 0 | - |
| L2 也算错了 | ❌ | ❌ | ✅ 滑点保护 |

→ 三层都失效的概率 ≈ 0，比单层风控可靠 100 倍。

## 7. 部署形态

### 7.1 物理部署

单机部署，所有组件跑在同一个 VPS / 家用服务器：

```
┌─────────────────────────────────────────┐
│  systemd 守护                            │
│  ├── crypto-quant-data       (数据采集) │
│  ├── crypto-quant-engine     (策略+风控)│
│  ├── crypto-quant-executor   (执行)     │
│  ├── crypto-quant-dashboard  (监控)     │
│  └── crypto-quant-reporter   (日报)     │
│                                          │
│  共享:                                   │
│  ├── data/crypto.sqlite      (主库)     │
│  ├── data/candles/           (parquet)  │
│  └── config/*.yml                        │
└─────────────────────────────────────────┘
```

每个组件独立进程，挂掉自动重启，相互不影响。

### 7.2 进程间通信

不引入 MQ（避免复杂度）。三种通信方式：
1. **数据库**：跨进程持久化数据
2. **共享内存（python multiprocessing.Manager）**：高频热数据（最新价格、当前持仓）
3. **本地 HTTP**：Dashboard 调 engine 状态接口

### 7.3 配置管理

- `config/config.yml` 主配置
- `config/secrets.yml` API key（gitignore，权限 600）
- `config/strategies/sX.yml` 每策略一份参数
- 配置**热加载**：改配置不重启进程

## 8. 演进路线

按里程碑演进，不一次到位。

### Phase 1：基础设施（2-3 周）
- 数据层 + 历史回填 + 回测引擎
- paper 撮合器
- Dashboard 最小版

### Phase 2：研究区完善（2 周）
- 因子库 ≥ 10 个
- Walk-forward + OOS 框架
- 评分卡自动化

### Phase 3：策略 + 实盘 OMS（2-3 周）
- 3 个种子策略实现
- 实盘下单（含强制止损）
- 三层风控完整化

### Phase 4：上线 + 迭代（持续）
- 单策略小资金试水
- paper-vs-live 偏差监控
- 季度新策略孵化

详细任务和里程碑见 [`08-路线图.md`](./08-路线图.md)。
