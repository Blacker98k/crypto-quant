## 1. 项目背景

维护者 Blacker 已积累自动化交易系统的工程经验：多线程数据采集、信号生成、三层风控熔断、配置热加载、Dashboard 监控、LLM 辅助打分等模块均有实战。本项目希望复用这些工程能力，做一个**虚拟币量化交易**系统。

**币圈相比早期实践场景的优势**：
- 市场**连续**且 24/7 运行，可做组合管理而非离散事件下单
- 流动性**充足**，可双边开仓 / 套利
- 有完整的**历史数据**可回测，能在实盘前充分验证
- 标的丰富，可做风格独立的多策略组合

**币圈相比早期实践场景需要额外重视的点**：
- 杠杆 + 24/7 → 风险被**放大**，必须三层风控冗余
- 行情连续 → 不挂交易所侧止损，宕机即可能爆仓
- 历史数据齐全 → **没有理由跳过回测**

## 2. 核心目标与非目标

### 2.1 核心目标（Must Have）

| # | 目标 | 衡量 |
|---|---|---|
| G1 | 多策略并行运行 | 同时运行 ≥ 3 个风格独立的策略 |
| G2 | 研究-生产代码一致 | 同一份 `Strategy` 代码可在回测和实盘运行 |
| G3 | 实盘前必须 paper 验证 | 任何策略上实盘前 paper 跑 ≥ 30 天 |
| G4 | 强制风险熔断 | 单日亏 5% 自动停 / 总回撤 20% 强制清仓 |
| G5 | 7×24 自动运维 | 服务挂了仍有交易所侧止损保命 |
| G6 | 可监控 | 实时 Dashboard + 关键事件 Telegram 推送 |

### 2.2 非目标（明确不做）

| 项 | 原因 |
|---|---|
| 高频做市 / HFT | 散户延迟和返佣比不过专业团队 |
| 山寨币内幕交易 / 拉盘 | 不符合"系统化"原则 |
| 跟单 / 抄机器人 | 没有 alpha 来源 |
| DeFi 链上策略 | gas/MEV/钱包安全复杂度太高，先不碰 |
| 复杂衍生品（期权、双币） | 永续以外的合约暂不考虑 |
| 跨多交易所 | 第一阶段单交易所先跑通 |
| 合规出入金 | 项目假设 USDT 已在境外交易所 |

> 这些假设直接决定后续架构选型。Review 时不同意哪条直接改。

| # | 假设项 | 默认值 | 影响 |
|---|---|---|---|
| A1 | 起始基准 | 不在仓库文档中记录具体金额；由本地配置和风控参数决定 | 仓位算法、单次风险 |
| A2 | 标的范围 | **现货 + 合约（U 本位永续）** | 是否需要做空、对冲 |
| A3 | 杠杆上限 | **3x（合约部分）** | 强平距离、风险算法 |
| A4 | 周期偏好 | **中频（15m-4h），少量日线** | 不做高频，数据频率适中 |
| A5 | 收益目标 | 不在仓库文档中记录固定收益数字；以长期 paper / OOS 证据评估 | 风险预算、策略选型 |
| A6 | 回撤约束 | 不在仓库文档中记录具体阈值；以风控配置和复盘结论为准 | 熔断阈值 |
| A7 | 交易所 | **Binance（主）+ OKX（备）** | API 适配、数据源 |
| A8 | 运维强度 | **7×24 自动 + 强制交易所侧止损** | 系统可靠性等级 |
| A9 | 入金渠道 | 假设已有 USDT 在境外账户 | 不做合规出入金 |
| A10 | 时间预算 | **2-3 个月成型，半年迭代** | 第 1 月仅基础设施 |

## 4. 用户角色

### 4.1 维护者（Blacker）

- 唯一开发者 + 唯一资金持有人
- 工作日晚上 + 周末投入
- 编程基础有，不是专业量化背景
- 风险偏好：中等（不接受 30%+ 回撤）

### 4.2 AI Agent（多个，接力开发）

- 不同设备、不同会话的 AI 编程助手
- 必须能通过文档快速 onboarding（详见记忆库设计）
- 不应推翻已讨论过的决策（详见 ADR）

### 4.3 系统自身（自动化角色）

- **数据采集器**：7×24 拉行情数据
- **策略引擎**：定时评估信号
- **风控引擎**：每笔订单前检查
- **执行引擎**：实盘下单 + 持仓管理
- **监控**：Dashboard + 异常推送

## 5. 业务目标与 KPI

### 5.1 交易表现 KPI（实盘后衡量）

| 指标 | 目标 | 说明 |
|---|---|---|
| 年化收益率 | 配置化 | 只作为长期 paper / OOS 后的观察项 |
| 最大回撤（MDD） | 配置化 | 超过风控阈值后强制停机复盘 |
| Sharpe Ratio | 配置化 | 只作为策略比较指标，不写个人收益目标 |
| Calmar Ratio | 配置化 | 只作为策略比较指标，不写个人收益目标 |
| 月度盈利月数 | 配置化 | 不要求每月盈利，重点看长期稳定性 |

### 5.2 工程 KPI

| 指标 | 目标 |
|---|---|
| 系统可用性 | ≥ 99.5%（每月停机 ≤ 3.5h） |
| 异常响应时间 | 关键事件 < 1 分钟推送 |
| 回测速度 | 单策略 1 年 1m 数据 ≤ 1 分钟 |
| paper-vs-live 偏差 | < 5%（同期收益对比） |

## 6. 功能需求（高层次）

只列**做什么**，不讲**怎么做**（怎么做在架构和详设里）。

### 6.1 数据层

- F1.1 接入交易所行情 API，订阅 K 线、订单簿、成交、资金费率
- F1.2 历史数据回填（首次启动拉 ≥ 2 年数据）
- F1.3 数据持久化、可被研究和生产共用
- F1.4 断线自动重连 + 数据补洞

### 6.2 研究区

- F2.1 因子库：常用技术因子 + 资金面因子
- F2.2 回测引擎：向量化（快速）+ 事件驱动（准确）双引擎
- F2.3 策略评分卡：Sharpe / Calmar / MDD / 胜率 / 相关性自动出报告
- F2.4 Walk-forward 验证：滚动窗口防过拟合
- F2.5 OOS 验证：训练 60% / 验证 20% / **测试 20% 只用一次**

### 6.3 策略层

- F3.1 统一 `Strategy` 接口，研究和生产共用
- F3.2 多策略并行，组合管理（信号去重、相关性控制）
- F3.3 第一阶段 3 个种子策略，详见 [`docs/03-详细设计/strategies/`](../docs/03-详细设计/strategies/)：

| 编号 | 策略 | 风格 | 标的 | 周期 | 杠杆 | 评估方式 |
|---|---|---|---|---|---|---|
| S1 | BTC/ETH 趋势跟随 | 趋势 | BTC/ETH 永续 | 4h-1d | 低杠杆 | 回测 + OOS + paper 观察 |
| S2 | 中市值均值回归 | 反转 | top 50 永续（除 BTC/ETH） | 1h-4h | 低杠杆 | 回测 + OOS + paper 观察 |
| S3 | 资金费率套利 | 中性套利 | top 30 永续 + 现货 | 8h | 低杠杆 | 回测 + OOS + paper 观察 |

**为什么 3 个不是 10 个**：见 [ADR-005](../docs/07-决策日志/ADR-005-起步3个策略而非10个.md)。

### 6.4 币池分层（Universe Stratification）

不是"几百种币随便挑"，而是**分层管理**。每层的资源优先级、可选策略、监控频率不同：

| 层级 | 数量 | 资源优先级 | 适用策略 | 监控频率 |
|---|---|---|---|---|
| **核心池** | top 20 | 高 | 全部策略（S1/S2/S3） | 实时 WS |
| **观察池** | top 21-100 | 中 | S2 / S3（S1 不做） | 1m K 线 |
| **普查池** | top 101-300 | 低 | 仅 S3（套利）/ 日级扫描 | 1h K 线 |
| **黑名单** | 上线 < 30 天 / 流动性差 | 排除 | 完全排除 | - |

**核心池筛选规则**（top 20）：
- 按 24h 成交额排序，取 top 20
- 排除稳定币（USDC/DAI 等）
- 排除绑定币（如 BUSD/BNB 这种受单一交易所影响）
- 每周一更新一次币池

**观察池筛选规则**（top 21-100）：
- 按 24h 成交额排序，取 top 21-100
- 排除上线 < 30 天的新币
- 排除"meme 币"（如 DOGE 这种由社交媒体驱动）—— **可选规则**，待回测验证后决定
- 每周一更新

**普查池筛选规则**（top 101-300）：
- 用于发现新的资金费率套利机会
- 不做趋势/反转策略（流动性不足、回测可信度低）
- 每月更新

**为什么分 3 层**：

- 不分层 = 几百种币同时跑 = 数据/资源/运维爆炸
- 单层（只做 top 20）= 错过山寨币的套利机会
- 分 3 层 = 平衡分散度与运维成本
- 资源按优先级分层，核心池优先实时监控，普查池仅用于机会发现

详见 [详细设计/策略层](../docs/03-详细设计/02-策略层.md)（待编写）和 [Strategy Manager 组合管理](../docs/02-架构设计文档.md#35-strategy-manager组合管理器)。

### 6.5 风控层

- F4.1 三层风控：投资组合层 / 仓位层 / 订单层（详见 [ADR-008](../docs/07-决策日志/ADR-008-三层风控冗余.md)）
- F4.2 熔断：阈值全部配置化，不在仓库文档中记录个人资金口径
- F4.3 仓位算法：ATR-based，单笔风险由配置和风控模块决定
- F4.4 **强制交易所侧止损单**（详见 [ADR-003](../docs/07-决策日志/ADR-003-强制交易所侧止损.md)）

### 6.6 执行层

- F5.1 OMS：限价 / 市价 / 止损 / 止盈 / IOC / FOK
- F5.2 部分成交追单
- F5.3 滑点监控
- F5.4 失败重试 + 幂等（client order id）

### 6.7 监控层

- F6.1 实时 Dashboard：持仓 / PnL / 风险事件 / 策略表现
- F6.2 Telegram bot 推送：开仓 / 平仓 / 熔断 / 异常
- F6.3 每日报告：当日交易、盈亏、风险事件汇总

### 6.8 paper 模式

- F7.1 与实盘共用代码，仅最后下单步骤不真发
- F7.2 模拟撮合：含手续费 + 滑点
- F7.3 paper 数据独立存储，不污染实盘数据

## 7. 非功能需求

### 7.1 性能

- 数据延迟：WS 行情 → 策略可见 < 1s
- 回测速度：100 万根 K 线 < 1 分钟（向量化）
- Dashboard 响应：< 500ms

### 7.2 可靠性

- 系统可用性 ≥ 99.5%
- 数据持久化无丢失（事务 + WAL）
- 服务挂掉仍有交易所侧止损保命

### 7.3 可观测性

- 所有重要事件落库（risk_events 表）
- 结构化日志（JSON）
- 健康自检接口

### 7.4 安全

- API key 不入库 / 不入 git，仅放 `secrets.yml` + 600 权限
- 实盘前两阶段确认（paper → small live → full live）
- 关键操作（清仓/熔断）需要可审计

### 7.5 可维护

- 代码模块化，每模块 ≤ 500 行
- 文档与代码同步更新
- 多 Agent 接力友好（记忆库 + ADR）

## 8. 关键约束

| 类型 | 约束 |
|---|---|
| 资源 | 单机部署（家用服务器 / VPS） |
| 网络 | 国内访问 Binance 需稳定代理 |
| 时间 | 维护者业余时间投入 |
| 资金 | 仓库不记录具体金额；paper-only 观察期和实盘准备按风控配置执行 |
| 法律 | 项目仅限个人使用，不对外提供服务 |

## 9. 验收标准

按 Phase 验收（详细路线见 [路线图](./08-路线图.md)）：

### Phase 1：基础设施（2-3 周）
- ✅ 能拉历史 K 线 + 订阅实时 WS
- ✅ 能用 vectorbt 跑通 BTCUSDT 双均线 demo，输出报告
- ✅ Dashboard 最小版能看到 K 线和持仓

### Phase 2：研究区（2 周）
- ✅ 因子库 ≥ 10 个常用因子
- ✅ Walk-forward 框架可用
- ✅ 自动出 HTML 评分卡

### Phase 3：策略 + OMS（2-3 周）
- ✅ 3 个策略实现并通过 OOS 验证
- ✅ 实盘 OMS 含强制交易所侧止损
- ✅ 三层风控全部生效

### Phase 4：上线（持续）
- ✅ 选 1 个策略进入小额实盘准备流程；具体金额不写入仓库
- ✅ paper-vs-live 偏差 < 5%
- ✅ 观察期后按复盘结论决定是否扩大范围

## 10. 风险与限制

详见 [决策日志 ADR](./07-决策日志/) 和 [踩坑记录](./09-踩坑记录.md)。

### 系统性风险
- 交易所暴雷（FTX 模式）→ 资金分散、留 50% 在冷钱包
- API 封号 / 限流 → 严守 rate limit、用 sub-account
- 黑天鹅行情（312/519）→ 杠杆 ≤ 3x + 强制交易所侧止损

### 策略风险
- 过拟合 → walk-forward + OOS + paper 30 天
- regime change → 多策略组合
- 流动性陷阱（山寨币）→ 严格选币池

### 工程风险
- 数据不一致（回测 vs 实盘）→ paper 撮合器复用
- 状态不一致（本地 vs 交易所）→ 5min 对账
- 风控 bug → 三层冗余 + risk_events 落库

### 心理风险
- 手痒干预 → 系统起来后绝不手动操作
- 过早扩本金 → 30 天观察期硬性规定
- 怕亏不看 → Telegram bot 强制推送
