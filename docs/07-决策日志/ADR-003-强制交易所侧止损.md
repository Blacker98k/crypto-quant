# ADR-003 · 任何持仓必须有交易所侧止损单

> **状态**：已生效（不可妥协）
> **决定时间**：2026-04-29
> **影响范围**：执行层、风控层
> **关联**：[ADR-008 三层风控冗余](./ADR-008-三层风控冗余.md)

---

## 1. 背景

币圈合约持仓与传统离散下单的关键差别：价格 24/7 连续变动，杠杆持仓会随时被强平。一旦持仓没有交易所侧的硬保护，本地服务任何一种"停摆"都可能在几分钟内吃掉本金。

实际可能的灾难场景：
- 服务器宕机 / VPS 重启 / 内存溢出
- 网络异常 / 代理挂掉 / DNS 故障
- 程序 bug 卡死循环 / 死锁
- API rate limit 触发 / IP 被封
- 维护者睡着或外出无信号

维护者已在过往项目中实际遇到过类似的"静默故障"：
- HTTP 客户端连接池长时间腐化，请求发不出去但程序不感知
- 平台 token 静默失效，所有调用都返回错误但本地报错信息误导
- 代理挂掉时所有外部 API 不可达，但程序不退出也不告警

**币圈如果发生类似情况而本地止损未挂到交易所**，黑天鹅行情可以在几分钟内吃掉本金。

---

## 2. 决定

**所有合约持仓必须在交易所侧挂止损单（stop-loss order）**。

具体规则：
- 开仓后 **5 秒内**必须挂止损（开仓+挂止损视为原子操作）
- 5 秒内未挂上 → **自动平仓**该持仓（不允许"无保护"持仓）
- 止损价 = 入场价 - 1.5 × ATR（多头），方向相反对空头
- 止损单挂在**交易所**，不在本地内存
- 持仓状态变化（部分成交、加仓）时止损单需要相应调整

---

## 3. 备选方案

### A. 仅本地止损（程序持续监控价格触发）
**优点**：止损价更灵活（可以跟踪止损、ATR 动态调整）。  
**缺点**：
- 程序挂了止损不生效 → 致命缺陷
- 网络断了无法下单
- API 限流时下不出止损

### B. 仅交易所侧止损（本 ADR）
**优点**：交易所 24/7 监控价格，触发后必然成交。  
**缺点**：
- 止损价相对固定（动态调整需要撤销+重挂）
- 极端行情可能穿透止损（slippage）
- 可能被"插针"扫掉一些利润

### C. 双层止损（A + B）
**优点**：理论上最稳。  
**缺点**：实现复杂，两个止损可能冲突触发，工程成本高。**单兵作战不现实**。

---

## 4. 选择理由

**关键权衡**：极端行情少见但致命 vs 偶尔被插针扫单。

期望值计算：
- 黑天鹅每年 2-3 次（312、519、FTX、银行危机等）
- 单次黑天鹅+断网+无止损 → 可能损失 50-80% 本金
- 插针扫单频率：每月 0-2 次，单次损失 ≤ 1%（因为本就是止损位）

**前者期望损失** = 2.5 × 65% = **162.5% 年损失**（夸张但说明问题）  
**后者期望损失** = 18 × 0.5% = **9% 年损失**

→ 选 B 显然更合理。**这条不是"要不要"的问题，是"必须"**。

---

## 5. 后果

### 正面
- 服务器宕机仍然有保护
- 维护者可以放心睡觉/出差
- 黑天鹅行情有兜底
- 心理压力大幅降低

### 负面
- 部分利润被插针吞掉（统计上可接受）
- 跟踪止损实现复杂（需要撤销+重挂）
- 交易所侧止损可能在特殊情况下不生效（如交易所宕机本身）

### 中性
- 需要监控止损单的状态（确保挂上、未被错误取消）
- 止损单也会消耗 API rate limit

---

## 6. 何时重新审视

**永不**。这是不可妥协的红线。

例外条款：
- 现货持仓（非杠杆）可以不强制（无强平风险）
- 资金费率套利的对冲对（多空对开）可以延后挂止损（基差风险有限）
