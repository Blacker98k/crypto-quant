# ADR-004 · 单交易所（Binance）先跑通，不一开始多交易所

> **状态**：已生效
> **决定时间**：2026-04-29
> **影响范围**：交易所适配层、Phase 4 之前
> **关联**：[ADR-001 自研框架](./ADR-001-自研框架.md)

---

## 1. 背景

币圈量化做多交易所的好处：
- 跨所套利机会（基差、价差）
- 单所故障的容灾
- 资金分散降低暴雷风险（FTX 模式）

但**多交易所工程量翻倍**：
- 每家 API 风格不同（REST + WS 都要适配）
- 鉴权方式不同（HMAC、Ed25519、API key 头不同）
- 限流策略不同（Binance 1200 req/min、OKX 不一样）
- 时区/时间精度不一致
- 资金管理需要跨所对账
- 手续费体系不同

需要决定 Phase 1-3 是单交易所还是多交易所。

---

## 2. 决定

**Phase 1-3 仅支持 Binance。OKX 等接入推迟到 Phase 5+。**

具体规则：
- 数据层、OMS、Strategy 全部针对 Binance 优化（不做"通用 adapter"幻想）
- 但代码组织上**预留多交易所扩展点**（如 `core/data/exchanges/binance.py` 而非直接写 `core/data/feed.py`）
- 配置 `exchange: binance` 字段保留（即使只有一种值）

---

## 3. 备选方案

### A. 单交易所先做（本 ADR）
**优点**：工程量减半，1 家做透。  
**缺点**：错过跨所套利机会。

### B. 一开始就支持多交易所
**优点**：长远不用重写。  
**缺点**：
- Phase 1-3 时间至少翻倍
- 散户单兵作战不现实
- 多账户监控/对账增加运维负担

### C. 先做 ccxt 通用层（不绑定具体交易所）
**优点**：理论上"换一行代码就能换交易所"。  
**缺点**：
- ccxt 的统一接口在订单簿、止损单、保证金类型上**有大量交易所差异**
- 实际上一旦遇到具体边界情况就要写交易所专用代码
- "通用层"是工程界经典 trap：写得越通用，对每个具体场景都越难

---

## 4. 选择理由

**关键权衡**：未来灵活性 vs 当下开发速度。

Binance 一家覆盖前 6 个月需求的依据：
- 流动性：BTC/ETH 现货+永续 全球前列
- 币种数：top 300 全部覆盖
- API 稳定：年宕机时间 < 1 小时
- 文档：齐全（虽然中文翻译不完整）
- 资金费率套利：核心标的全在 Binance
- ccxt-pro 对 Binance 支持最完整

**多交易所是 Phase 5+ 的"奢侈品"**：等本项目稳定盈利 6 个月后再考虑。

---

## 5. 后果

### 正面
- Phase 1-3 工期可控（2-3 个月内完成）
- 代码量小，易测试
- 集中精力解决一家的细节问题

### 负面
- **错过跨所套利**（基差套利等）—— 这是真正的 alpha 来源之一
- Binance 暴雷或 API 重大变更时无备份
- 资金 100% 在 Binance（暴露单点风险）

### 中性
- 资金分散问题可以**通过冷钱包**部分解决，具体比例不写入仓库
- 后续接 OKX 时需要重新设计 adapter（但有 Binance 经验做参考）

---

## 6. 何时重新审视

满足以下任一条件时启动多交易所计划：

- Phase 4 实盘连续盈利足够长时间，且资金规模达到维护者线下设定的迁移阈值
- 发现 Binance 上无法实现的策略（如某 OKX 独占的资金费套利机会）
- Binance API 重大变更或合规风险
- 团队从单兵作战扩大到 2+ 人
