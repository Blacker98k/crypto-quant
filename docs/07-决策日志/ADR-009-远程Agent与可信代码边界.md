# ADR-009 · 远程 Agent 与可信代码边界

> **状态**：已生效
> **决定时间**：2026-04-29
> **影响范围**：协作流程、Git 仓库权限、安全
> **关联**：[ADR-001 自研框架](./ADR-001-自研框架.md) · [ADR-006 文档中文命名+渐进式披露](./ADR-006-文档中文命名渐进式披露.md)

---

## 1. 背景

本项目是单人维护 + 多 AI Agent 接力开发：

- **核心维护者**：Blacker，物理上唯一可访问主分支的人
- **会话型 Agent**（Cursor / Claude / GPT 等内嵌在 IDE 中的）：每次会话临时启动，本身不持有 git 凭据，所有写入都经过维护者本地工具链
- **常驻 Agent**（如 OpenClaw 这类 7×24 在线 Agent）：独立部署在维护者本机，**拥有本机 shell / 文件系统能力**（能改任何文件、跑任何命令），但**默认不持有远程仓库 push 凭据**——可以在本机生成 diff / patch 文件，**是否本机 commit 也由维护者确认**（避免"看似临时"的工作不知不觉就进了 git 历史）

部署形态决定了"权限边界"问题尤为突出：

- 项目部署在维护者**本地 Linux 物理机**上长期运行（开发机即生产机）
- 常驻 Agent 与项目代码、运行时、数据库、API key 同机
- Agent 即使出于善意改了"看似没问题"的代码，也可能在边缘情况下让风控失效、强平、资金损失

量化交易系统涉及真金白银。**任何代码改动必须经维护者本人审阅才能合入主分支**——这是不可妥协的红线。

---

## 2. 决定

**主分支（`main`）写权限仅限维护者本人**。所有 Agent（含会话型与常驻型）的产出必须经 PR + 人审才能合入。

### 2.1 工作产出与权限分层

不同类型的 Agent 有不同的"默认产出位置"。**越接近 main，门槛越高**：

| 产出位置 | 谁能写 | 适用 | 进入下一级的条件 |
|---|---|---|---|
| **本地工作目录 / patch 文件** | 任何 Agent（含常驻 OpenClaw 类） | 默认 | 维护者本人 review 后才允许 |
| `agent/<agent-name>/<task>` 远程分支 | 对应 Agent，且需维护者**显式开启** | 已通过初步 review，需要远程协作 / CI 跑测时 | PR + 人审 |
| `feat/*` / `fix/*` 分支 | 维护者或 Agent（经审后） | 准备并入 main 的迭代分支 | PR + 人审 + CI 通过 |
| `main` | 仅维护者本人 | 唯一稳定分支，部署源 | —— |

**OpenClaw 这类常驻 Agent 的默认行为**：

- 默认**只产出本地 patch / 工作目录文件**（即"临时性工作"），**不**直接 push 到任何远程分支
- 维护者在本机 review 后，可以选择：
  - 丢弃（最常见）
  - 转成 git diff / commit 到 `agent/openclaw/<task>` 远程分支（继续 PR 流程）
  - 维护者本人接手改造后用自己身份提交
- **常驻 Agent 不持有任何远程仓库的 push 凭据**（即使本机能 push，仓库侧分支保护也会拒）

> 这是有意设计的"双层闸门"：第一层（本机 review）拦掉绝大多数临时探索；第二层（PR 人审）拦掉看似 OK 但实际有风险的实现。

### 2.2 审阅强制项

合入 main 前必须满足：

- [ ] 经过至少一次 PR Review（维护者本人）
- [ ] CI 通过（lint / 单测 / 关键集成测）
- [ ] 涉及风控 / OMS / 实盘代码的改动需要更严格的逐行审
- [ ] 涉及 secrets / API key 配置的改动**禁止 Agent 直接动**，必须维护者本人改

### 2.3 高敏感文件白名单（Agent 不得直接修改）

| 类别 | 路径示例 | 严格度 |
|---|---|---|
| 凭据 | `config/secrets.yml`、`.env`、任何含 API key 的文件 | **绝对禁止**——必须维护者本人改 |
| 风控阈值 | `config/risk.yml`（单笔风险 / 杠杆上限 / 熔断阈值等） | **绝对禁止**——必须维护者本人改 |
| 实盘开关 | `config/runtime.yml` 中的 `mode: live`、二阶段确认开关 | **绝对禁止**——必须维护者本人改 |
| Git 历史 | `.git/` 目录、`git rebase / push --force` | **绝对禁止** |
| 部署脚本 | `scripts/deploy.sh`、`systemd` unit 文件 | **绝对禁止** |
| `.gitignore` | 项目根目录的 `.gitignore` | **有限允许**：详见 §2.3.1 |

#### 2.3.1 .gitignore 的特殊规则

`.gitignore` 既是工程基础（项目早期需要 Agent 协助起草），又决定"什么会被推到远程"（敏感）。区分两种修改：

- ✅ **Agent 可提 PR**：初始创建、添加显然应忽略的产物（`__pycache__`、`*.pyc`、`*.log`、`data/`、`reports/`）
- ❌ **Agent 不得自动改**：增删 `secrets.yml` / `.env` / `config/*.yml` 等"可能泄露凭据或风控配置"的条目

判断标准：**该条目的存在与否会不会改变敏感数据的入库范围**。会改变 → 必须维护者本人改。

Agent 想改 §2.3 中"绝对禁止"类的文件 → 在 PR 描述里**显式声明缘由**，由维护者本人接手修改后再合（不允许"Agent 给草稿 + 维护者点合并"这种流程，必须维护者亲手改）。

### 2.4 Agent 产出的可追溯性

每次 Agent 提交必须显式标注：

```
[agent: <agent-name>] <commit message>

Generated by: <agent-name>
Session: <session-id 或时间戳>
Task: <用户给出的原始任务摘要>
Reviewed by: <pending 或 维护者签名>
```

例如：

```
[agent: openclaw] 实现 ATR 仓位算法

Generated by: openclaw
Session: 2026-05-15-14:23
Task: 按 docs/03-详细设计/03-风控层.md §4 实现 Position Sizer
Reviewed by: pending
```

## 3. 备选方案

### A. 完全禁止 Agent 写权限
**优点**：绝对安全。  
**缺点**：常驻 Agent（OpenClaw）的价值大幅打折——它本来就是为了在维护者不在线时持续推进。

### B. 给 Agent push 权限不做审阅
**优点**：开发节奏快。  
**缺点**：
- 任何 bug 直接进入主分支，可能影响生产
- 风控代码被无意改坏 = 资金损失
- 散户单兵作战，没有第二人兜底

### C. PR + 人审 + 高敏感文件白名单（本 ADR）
**优点**：
- Agent 有产出能力但有审查门控
- 风控 / 凭据 / 实盘开关绝不被 Agent 单独修改
- 提交历史可溯源到具体 Agent 与任务
**缺点**：维护者必须按时审 PR，否则 Agent 工作排队堆积

---

## 4. 选择理由

**关键权衡**：常驻 Agent 价值（持续推进）vs 资金安全（合入审查）。

资金安全无条件优先。理由：

- Agent 的"判断力"在金融场景没有可信的第三方校准
- 风控失效一次的损失 ≫ 多次 Agent 工作排队的延迟成本
- 维护者本人审 PR 是对系统状态的**强制再认知**，避免"代码进生产但维护者根本不知道"

**人审的额外好处**：
- 强迫维护者每隔一段时间复盘代码状态（回到"驾驶员位置"）
- 避免"自动化幻觉"——Agent 改了一堆但没人真的理解改了什么

## 5. 后果

### 正面

- main 分支永远是经过维护者认知的状态
- 资金 / 凭据 / 风控阈值有"人闸"
- Git 历史能追溯到"谁的责任"（具体 Agent + 任务）
- 维护者随时可以审计 Agent 的产出质量

### 负面

- Agent 工作可能排队等审
- 维护者审 PR 的工作量不会消失
- 维护者出差/休假时常驻 Agent 的产出会堆积

### 中性

- 需要在仓库里加分支保护规则（GitHub Settings → Branches）
- 需要 CODEOWNERS 文件
- 需要 commit message 模板（强制 Agent 标注）

---

## 6. 配套规则

### 6.1 仓库层（GitHub）

- 启用 `main` 分支保护：禁止直推、必须 PR、必须 review
- CODEOWNERS：所有路径默认 `@Blacker98k`
- secrets / 风控阈值文件加路径级保护（如有）

### 6.2 本地层（开发机）

- 常驻 Agent 默认**不持有远程 push 凭据**——只能在本地工作目录、patch 文件、未 push 的本地分支上活动
- 维护者本人在 review 后才决定是否把 Agent 的产出 push 出去（用自己的身份）
- 如有特殊场景需要 Agent 可 push（比如 CI 跑测），仅授予 `agent/<name>/*` 分支的推送权限，并用 `pre-push` hook 拦截其他分支
- secrets.yml 文件权限 `600`，仅维护者用户可读
- API key 不允许出现在任何被 git 追踪的文件里

### 6.3 文档层

- 项目根目录的 `AGENTS.md`（待写）说明：哪些 Agent 在用、各自的 scope、commit message 规范
- 维护者审 PR 时按本 ADR §2.2 checklist 逐条对

---

## 7. 何时重新审视

- 出现成熟可信的 LLM 代码自动审计方案（覆盖金融特化）→ 部分自动化审查
- 项目从单兵作战扩展到 2+ 维护者 → 重新设计 review 矩阵
- 出现具体的 Agent 误操作事件 → 加严规则

**永不取消**的部分：

- main 分支只能由人合入
- 高敏感文件白名单
- Agent 产出必须可追溯
