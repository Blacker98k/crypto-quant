# ADR-006 · 文档全部中文命名 + 渐进式披露

> **状态**：已生效
> **决定时间**：2026-04-29
> **影响范围**：所有 docs/、记忆/、skills/ 目录组织
> **关联**：（无强依赖）

---

## 1. 背景

启动文档体系时面临两个独立但相关的决策：

**问题 1：文档命名是英文还是中文？**
- 英文：工程界默认（kebab-case），跨工具兼容性好
- 中文：维护者母语，查找直观

**问题 2：长文档 vs 拆分？**
- 长文档：信息集中，单文件查找方便
- 拆分（渐进式披露）：每文件聚焦，LLM 上下文不爆炸

---

## 2. 决定

### 2.1 中文命名

- **文档文件名用中文**（如 `01-产品需求文档.md`）
- **目录名用中文**（如 `记忆/`、`docs/03-详细设计/`）
- **代码（Python）文件名用英文 snake_case**（避免 import 路径问题）
- **配置文件用英文 kebab-case**（如 `s1-btc-trend.yml`）
- **数据文件用英文**（parquet/sqlite 文件路径）

### 2.2 渐进式披露

- **每份文档 ≤ 500 行**（一屏读完，LLM 上下文友好）
- **入口/索引文档 ≤ 100 行**（README、记忆/README）
- **多文档拆子目录**（详设/接口/ADR/skills 都是子目录）
- **文档开头 80 字摘要**（LLM 先扫摘要决定是否展开）
- **不重复内容，互相链接**（用相对路径）

---

## 3. 备选方案

### 命名方案

#### A. 全英文（业界标准）
**优点**：跨工具兼容、分享给国外人没障碍。  
**缺点**：维护者中文母语，查找慢；翻译成本（如 "Decision Log" vs "决策日志"）。

#### B. 全中文（本 ADR）
**优点**：查找快、表达精确。  
**缺点**：少数工具/CI 上中文路径有 bug（罕见）；分享成本（但本项目私有）。

#### C. 混合
**优点**：兼顾。  
**缺点**：风格不统一，决策成本高（每个文件都要决定）。

### 拆分方案

#### A. 单一长文档
**优点**：一处查所有。  
**缺点**：
- LLM 单次读完爆 context（5000+ 行不现实）
- diff 时改一小处全文都被标记
- 多人协作冲突点多

#### B. 渐进式披露（本 ADR）
**优点**：单文件聚焦、LLM 友好、diff 清晰。  
**缺点**：跳转成本（但有相对链接）；初期写起来稍累（要想清楚拆分边界）。

---

## 4. 选择理由

### 中文命名的理由

**用户明确要求**——这是用户偏好，不需要技术理由。

**附加好处**：
- 项目是私有项目，不需要给国外人看
- 维护者 + AI 协作场景下，AI 理解中文文件名无障碍
- git/Linux 文件系统对 UTF-8 中文支持良好

### 渐进式披露的理由

**多 Agent 接力是核心场景**：
- 新 Agent 进来不该被迫读 5000 行才知道项目状态
- 应该 5 分钟内完成 onboarding（读 README + 记忆/README + 项目概览）
- 具体任务才查对应详设/接口/ADR

**LLM 上下文成本**：
- Claude / GPT 单次输入 200k token 看似充足，但**上下文越长效果越差**
- 拆分让每次只加载 200-500 行真正相关的内容

---

## 5. 后果

### 正面
- 维护者查找 + 阅读速度提升
- LLM 上下文使用效率高
- 多 Agent 接力 onboarding 流程清晰
- diff 时改动范围明确

### 负面
- 跨平台兼容性轻微风险（特别是 Windows + 老工具）
- 维护者切换到不支持中文输入法的设备时打字麻烦（虽然可以复制路径）
- 拆分边界判断需要思考（但写多了有套路）

### 中性
- 跟其他工程项目风格不一致（但本项目是个人项目，不需要统一）

---

## 6. 何时重新审视

- 项目从私有变公开 / 加入国外协作者
- 中文路径在某关键工具上出现 bug 且无法 workaround
- 单兵作战变多人团队，统一规范成本上升
