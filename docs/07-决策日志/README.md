## 写作模板

新决策按下面模板写一份独立 md：

```markdown
# ADR-NNN · 标题

> **状态**：已生效 / 草稿 / 已废弃
> **决定时间**：YYYY-MM-DD
> **影响范围**：架构 / 数据 / 接口 / 流程

## 1. 背景
（什么场景下需要做这个决定）

## 2. 决定
（一句话决定，明确）

## 3. 备选方案
- A. 方案 A：优缺点
- B. 方案 B：优缺点
- C. 方案 C：优缺点

## 4. 选择理由
（为什么选 X，关键权衡是什么）

## 5. 后果
- 正面：得到了什么
- 负面：付出了什么代价
- 中性：需要后续观察的

## 6. 何时重新审视
（什么条件下需要重新讨论）

## 7. 关联
- 相关 ADR
- 受影响的代码模块
- 相关文档
```

---

## 何时写一份新 ADR

满足以下任一条件时**必须**写：

- 影响多个模块的设计决定
- 选择了某个技术栈/库（且有备选）
- 引入了不可妥协的约束（如"必须 paper 30 天"）
- 否决了一个看似合理的方案（防止后续重提）
- 改变了原 ADR（新写一份"取代 ADR-XXX"）

---

## ADR vs 踩坑记录

| 类型 | 何时写 | 例子 |
|---|---|---|
| **ADR** | **决策点**，多选一时 | "用 vectorbt 还是 backtrader" |
| **踩坑** | **事后**，遇到具体 bug 时 | "httpx 连接池腐化导致请求失败" |

两者交叉时：先写踩坑修复，发现是架构问题再升级为 ADR。
